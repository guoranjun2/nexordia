# 1.13.3 Cloud View 云框范围优化

## 背景

在大图或低缩放视图中，`RECT` 和 `ROUND_RECT` 云框有时会圈出远大于实际节点内容的区域。典型现象是云框顶部或底部包含大量空白，只在底部附近真正包住了节点内容。

这类问题主要出现在子节点分布跨度较大、布局方向切换、或节点自身内容可见但子树布局占用较大时。

## 排查结论

`RectangleCloudView` 原先直接使用 `source.getInnerBounds()` 作为云框绘制范围。

`NodeView.getInnerBounds()` 在节点自身内容可见时返回的是整个 `NodeView` 的内部区域。这个区域包含子树布局占用和空白，不等价于“可见节点内容的最小包围框”。因此矩形云框会把布局空白也圈进去。

对比其他云形状：

- `ARC` 和 `STAR` 通过 `CloudView.getCoordinates()` 收集可见节点内容点集，再计算凸包。
- `CIRCLE` 使用自身的圆形 bounds 计算逻辑。
- `RECT` 和 `ROUND_RECT` 原先没有按实际可见内容重新计算 bounds。

## 本次优化

矩形云框改为基于实际可见内容计算范围：

1. 递归遍历 cloud 根节点及其子节点。
2. 只收集 `nodeView.isVisible()` 且 `nodeView.isContentVisible()` 的内容组件 bounds。
3. 将内容 bounds 转换到 cloud 根节点坐标系。
4. 使用 `RectangleCloudBounds` 合并所有内容矩形。
5. 保留原有按父子布局方向扩展 padding 的行为。

相关实现：

- `RectangleCloudView.getVisibleContentBounds()`
- `RectangleCloudView.getRectangle()`
- `RectangleCloudBounds.fromContents(...)`

## 小尺寸跳过绘制

参考 `MainView` 在低缩放下避免绘制过小节点内容的思路，`CloudView.paint(...)` 现在先判断 cloud 自身绘制 bounds。

如果 cloud 自身 bounds 满足以下任一条件，则跳过绘制：

- `width < 10`
- `height < 10`

注意这里判断的是 cloud 自身计算出来的 bounds，不是 `MainView` 或 `NodeView` 的组件宽高。

各形状的 bounds 来源：

- `ARC` / `STAR`：默认使用凸包 `getCoordinates().getBounds()`。
- `CIRCLE`：使用圆形 `getCircle().getBounds()`。
- `RECT` / `ROUND_RECT`：使用修正后的矩形云框 bounds。

## 简化节点文字回绘

低缩放下 `MainView.paintComponent(...)` 会走简化绘制路径，只绘制节点背景和边框，不再绘制完整文字。对带 cloud 的节点来说，这会导致 cloud 仍然可见，但中心节点文字消失，不利于在大图低缩放时识别云框代表的主题。

简化绘制阈值不写死在 `ZoomableLabel` 中，而是从 `freeplane.properties` 读取：

- `simplified_painting_max_zoom=0.2`
- `simplified_painting_max_width=10`
- `simplified_painting_max_height=3`
- `cloud_painting_min_width=10`
- `cloud_text_painting_min_width=50`
- `paint_cloud_text=true`
- `paint_cloud_for_invisible_node=true`
- `paint_cloud_title_for_invisible_node=false`
- `paint_cloud_image=true`
- `cloud_image_fixed_icon=info`
- `cloud_image_opacity=1.0`

读取失败或配置缺失时，代码使用同样的默认值作为 fallback。

这些选项在偏好设置中放入新的 `Painting` tab 下的 `Low Zoom Painting` 分组，避免混在通用行为设置里，也给后续其他渲染设置预留位置。`cloud_painting_min_width` 控制 cloud bounds 宽度小于阈值时不绘制云框；`cloud_text_painting_min_width` 控制 cloud bounds 宽度小于阈值时不绘制中心文字，用于减少低缩放拖动画面时大量小云框文字测量带来的 repaint 成本。

`paint_cloud_text` 控制低缩放下是否绘制 cloud 中心文字，默认开启。`paint_cloud_for_invisible_node` 控制 cloud 根节点没有 visible paint anchor 时是否仍然绘制 cloud 本体，默认开启。`paint_cloud_title_for_invisible_node` 单独控制 cloud 根节点没有 visible paint anchor 时是否仍然绘制 cloud 中心文字，默认关闭，使 cloud 本体和 cloud title 可以独立配置。

## 云框图片和标题解析

云框中心内容现在分成两条互不覆盖的来源：

1. 云框文字。
2. 云框图片。

文字来源规则是：

- 如果节点存在 `title` 属性，云框文字使用该属性值；空字符串也是有效值。
- 如果节点不存在 `title` 属性，云框文字使用节点 `MainView` 文本的第一行。
- 如果节点文本是 HTML，非图片 HTML 会先转成纯文本，再取第一行。

图片来源规则独立于文字来源：

- 只有节点 `MainView` 文本是 HTML 且包含 `<img src=...>` 时，云框图片才会启用。
- `title` 属性不会阻止图片解析。也就是说，`title` 属性和节点标题互斥，但云框文字和云框图片可以共存。
- `<img>` 匹配大小写不敏感，支持单引号、双引号和未加引号的 `src`。

图片绘制不走完整 Swing HTML view，而是只解析第一张图片的 `src`，再复用 `HtmlImageCache` 的加载和缩放缓存。这样可以避免低缩放下大量云框触发完整 HTML layout，同时仍然沿用原有图片缓存、异步加载和 repaint 合并逻辑。

图片绘制限制在 cloud bounds 内，按 contain 规则缩放：

- 图片宽度不超过 cloud bounds 宽度。
- 图片高度不超过 cloud bounds 高度。
- 宽高比保持不变。
- 超出 cloud bounds 的内容会被 clip 掉，不会扩大 cloud 布局。

绘制层级保持文字可读：

- 云框图片在 `PaintingMode.CLOUDS` 中绘制，位于节点和云框文字下方。
- 云框文字仍在 `PaintingMode.CLOUD_TEXTS` 中绘制，压在图片和简化节点之上。
- 带图片的 HTML 仍会提取纯文本标题，不会因为显示图片而忽略文字。

新增配置：

- `paint_cloud_image=true` 控制是否绘制云框图片。
- `cloud_image_fixed_icon=info` 控制哪个图标让图片云框固定显示。
- `cloud_image_opacity=1.0` 控制云框图片透明度，偏好设置范围是 `0.05` 到 `1.0`，避免用户把图片调到完全不可见。

当节点带有 `cloud_image_fixed_icon` 指定的图标，并且该节点确实是图片云框时，它被视为 fixed image cloud。fixed image cloud 会绕过简化绘制和 visible paint anchor 限制：即使 cloud 根节点当前不在视图区域内，云框本体、云框图片和云框文字仍然可以绘制。这用于把少量重要图片云框作为视图外仍可见的地图标记。普通图片云框仍遵守原有可见性和简化绘制限制。

透明度只在最终图片绘制时通过 `AlphaComposite` 应用，不参与图片加载、缩放和缓存 key 计算。默认值 `1.0` 不设置额外 composite，保持当前视觉和性能路径；低于 `1.0` 时只在图片绘制期间临时设置 composite，并在绘制后恢复原值。

`ZoomableLabel.hasVisiblePaintAnchor(...)` 使用 label bounds 转换到 `MapView` 坐标后与 `visibleRect` 相交判断。此前只检查四个角点，并且当 label 宽高大于 viewport 时直接返回 true；这会在 70% / 80% 这类较大缩放下把远离视口的大节点误判为可见，导致其 cloud 边缘出现在当前视口中。改为矩形相交后，大节点只有实际覆盖视口时才被认为可见。

进一步排查 2% 缩放下大量云框导致卡死的问题后，定位到 `CloudView.paintDecoration(...)` 的内层循环。原代码使用 `int j` 累加 `double distanceBetweenPoints`；Java 的复合赋值会把结果强制转换回 `int`。在极低缩放下，`STAR` 等云框的 `distanceBetweenPoints` 可能小于 1，导致 `j += distanceBetweenPoints` 后仍为 0，循环条件持续成立，EDT 进入死循环。修复方式是使用 `double j` 表达真实的几何距离，并把最小步长限制为 1 像素，避免亚像素采样带来无意义的高频绘制。

## ARC 云框平滑轮廓优化

`ARC` 云框原先沿凸包边按距离拆成许多小段，每个小段创建一个 `QuadCurve2D` 并单独执行 `fill` / `draw`。给子节点很多的父节点添加 `ARC` 云框后，拖拽地图会触发大量 repaint，这部分临时曲线对象和绘制调用会成为明显瓶颈。

优化后 `ArcCloudView` 不再进入基类的分段圆弧绘制循环，而是基于凸包点生成一条闭合的平滑 `Path2D`：

1. 以凸包顶点作为曲线经过点，避免平滑曲线切入凸包内部后漏包节点。
2. 每条凸包边只生成一条二次曲线，控制点沿边的外法线推出。
3. 整个 ARC 云框只构造一个 path，并只执行一次 `fill` 和一次 `draw`。

这样保留了圆润外观，同时绘制复杂度从“凸包边长相关的多段曲线”降为“凸包顶点数相关的一条曲线”。视觉上不再使用旧算法的随机小圆弧扰动，因此边缘会更稳定、更接近平滑包络线。

`ARC` 仍然需要曲线填充和描边，不会像 `RECT` / `ROUND_RECT` 那样接近图形库原生矩形绘制速度。为减少剩余开销，`CloudView.getCoordinates()` 在单个 `CloudView` 实例内缓存凸包坐标，避免 `getPaintingBounds()` 和实际绘制各自重复计算一次凸包；`ArcCloudView` 使用 `Path2D.Float` 构造屏幕绘制路径，避免在像素级渲染中使用不必要的 double 精度路径。

本次新增逻辑：

1. `MainView.isPaintingSimplified()` 暴露当前 MainView 是否会走简化绘制路径。
2. `CloudView.paint(...)` 在 cloud 形状绘制完成后检查源节点的 `MainView`。
3. 如果源节点 `MainView` 正在简化绘制，则在 cloud 自身 bounds 中央绘制该 MainView 的文字。
4. 文字颜色使用 `mainView.getForeground()`。
5. 字体继承 `mainView.getFont()` 的字体族和样式，并动态派生字号。
6. HTML 文字先通过 `HtmlUtils.htmlToPlain(...)` 转成纯文本。

文字必须限制在 cloud 自身 bounds 内：

- 字号不写死，按 cloud bounds 二分搜索可容纳完整文字的最大字号。
- 搜索目标是让文字宽度尽量接近填满 cloud 宽度，同时文字高度不能超过 cloud 高度。
- 如果最小可用字号下完整文字仍然超过 cloud 宽度，使用 `...` 按当前字体度量截断。
- `...` 本身也放不下时不绘制文字。
- 绘制时额外用 cloud bounds 设置 clip，防止实际渲染越界。

绘制层级必须保证 cloud 文字在最上层：

- cloud 形状仍在 `PaintingMode.CLOUDS` 中绘制，保持原有云框位于节点下方的效果。
- cloud 文字不在 cloud 形状 pass 中绘制，避免被后续简化节点背景覆盖。
- 新增 `PaintingMode.CLOUD_TEXTS`，在 `NODES`、`SELECTED_NODES` 和 `LINKS` 之后执行。
- `NodeView.paintCloudTextTree(...)` 递归绘制 cloud 文字，使外框文字压在简化子节点之上。

该绘制发生在单独的 cloud text pass 中，因此只补偿 cloud 根节点的当前 MainView 文字，不改变普通节点绘制路径，也不影响不带 cloud 的低缩放节点。

## 曾检查过的方向

曾考虑直接沿用 `NodeView.getInnerBounds()` 后再修正部分坐标，但该方法本身语义是布局内部区域，不适合作为“实际内容包围框”。继续在它上面补丁容易把布局细节和云框绘制规则耦合得更深。

最终选择提取 `RectangleCloudBounds`，让矩形云框的业务规则更清楚，也方便单独测试矩形合并和 padding 扩展逻辑。

## 后续观察

后续需要重点观察：

- 低缩放下大量 cloud 同屏时 repaint 是否更稳定。
- `RECT` / `ROUND_RECT` 在布局方向切换、隐藏节点、summary 节点、折叠节点附近的包围范围是否符合预期。
- `width < 10` 或 `height < 10` 跳过绘制是否会让极小 cloud 在某些缩放级别提前消失。
