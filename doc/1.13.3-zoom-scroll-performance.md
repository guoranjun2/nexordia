# 1.13.3 缩放与滚动渲染性能优化记录

## 背景

这轮优化从大地图缩放、滚动卡顿开始。测试图中包含大量节点、云框、图片、details、note、连接线和背景图。开启性能日志后，瓶颈主要集中在以下几类：

- 云框绘制，尤其是 arc cloud 的坐标、填充、描边、装饰和云框文字。
- 缩放时节点 update 过重，很多与缩放比例无关的内容被重复计算。
- 大量节点处于简易绘制时，滚动已经可以较流畅，但缩放仍会因为 update 和布局失效产生卡顿。
- 非自由节点和 details view 对布局刷新更敏感，过度削减 update 会出现布局错位。

## 性能观测工具

新增并扩展了 `PaintPerformanceMonitor`，用于分解 paint、layout 和 zoom 的耗时。

主要能力：

- paint frame 统计，按慢帧或固定间隔输出。
- task 统计，用于非 paint 路径，例如 zoom update 和 layout validate。
- 子项计数和耗时统计，例如 `cloud=31.26ms/221`。

覆盖的关键分类包括：

- 云框：`cloud`、`cloudCoordinates`、`cloudPath`、`cloudFill`、`cloudDraw`、`cloudDecoration`、`cloudText`、`cloudImage`、`cloudImageText`。
- 节点：`mainView`、`mainViewFull`、`mainViewSimplified`、`mainViewTiny`、`modeNodes`、`modeSelectedNodes`。
- 边和连接线：`edgeCreate`、`edge`、`linkCreate`、`linkDraw`。
- 缩放 update：`zoomUpdate`、`zoomAnchor`、`zoomBackground`、`updateAllNodes`、`nodeUpdate`、`nodeUpdateContent`、`nodeUpdateDetails`、`nodeUpdateViewer`。
- 图片：`lazyImage`、`bitmapImage`。
- 布局：`layoutValidate`。

使用方式：

```bash
-Dorg.freeplane.performance.paint=true
```

可选参数：

```bash
-Dorg.freeplane.performance.paint.interval=60
-Dorg.freeplane.performance.paint.slowFrameMs=20
```

## 云框相关优化

### 云框渲染配置

新增云框绘制开关，并把云框相关配置聚合到单独配置组。

新增默认配置：

```properties
paint_cloud=true
paint_cloud_image=true
paint_cloud_text=true
paint_cloud_for_invisible_node=true
paint_cloud_title_for_invisible_node=false
```

新增 `CloudPaintingOptions` 统一读取云框相关属性。

作用：

- 可整体关闭云框绘制，用于性能验证或低性能场景。
- 可分别控制云框图片、云框文字、不可见节点云框和不可见节点云框标题。
- 云框属性变化后触发视图刷新，避免修改云框类型、颜色后需要手动缩放才生效。

### 云框绘制行为

调整云框绘制判断，使云框渲染可以按配置跳过。

保留点：

- 默认仍绘制云框，兼容既有行为。
- 用户拖拽文字或图片时不做突然隐藏，避免视觉突兀。

已尝试但未作为最终方向的方案：

- 缩放或滚动中临时隐藏云框文字/图片。性能有收益，但交互突兀。
- 针对云框做短时缓存，缩放后延迟清理。思路可行，但后续主要瓶颈转向缩放 update 和 details 布局，暂未继续推进。

## 简易绘制调整

去掉了 “Simplified node painting maximum zoom” 对简易绘制的缩放率限制。

保留的判断改为主要依赖节点屏幕尺寸：

- `simplified_painting_max_width`
- `simplified_painting_max_height`

作用：

- 任意缩放率下，只要节点在屏幕上足够小，就可以进入简易绘制。
- 大图滚动时显著减少 full main view painting。

已尝试但撤回：

- 当节点进入简易绘制时跳过 edge 绘制。
- 该方案滚动收益明显，但 edge 的可见性语义更复杂，尤其是父节点可见、子节点不可见时仍可能需要边；因此暂时撤回，保留正常 edge 绘制。

## 缩放轻量 update 路径

### 问题

原始缩放路径会对每个节点执行完整 update，包含：

- shape
- edge
- cloud
- CSS
- font
- text color
- attribute
- details
- note viewer
- icons
- main text

其中大量计算和缩放比例无关，尤其是 `nodeUpdateContent`。在测试日志中，完整刷新会出现类似：

```text
nodeUpdateContent=47-77ms/1051
zoomUpdate=52-87ms
```

这会让缩放明显卡顿。

### 当前方案

新增 `UpdateCause.ZOOM` 轻量路径。

缩放时 `MapView.setZoom(...)` 调用：

```java
updateAllNodeViews(UpdateCause.ZOOM);
revalidate();
repaint();
adjustBackgroundComponentScale();
```

`NodeView.update(UpdateCause.ZOOM)` 分两类处理：

- 自由节点走 `updateForZoom()` 轻量路径。
- 非自由节点走 `updateFully(UpdateCause.ZOOM)`，保留布局相关刷新，但不触发内容更新。

### 自由节点轻量路径

自由节点缩放时只做：

- 清理子节点布局缓存。
- invalidate `NodeView` 和 `MainView`。
- 更新 zoom 相关 min/max width。
- 更新 details、note、image viewer 的缩放相关尺寸/失效状态。
- 对可见内容 revalidate/repaint。

关键目标是避免触发：

- `updateShortener(...)`
- `updateIcons()`
- `mainView.updateText(...)`

这些内容更新会进入 `nodeUpdateContent`，是缩放卡顿的主要来源之一。

### 非自由节点处理

非自由节点对树布局更敏感。最初尝试让非自由节点完整 `UNKNOWN` update，布局正确性更好，但性能明显退化。

问题日志表现：

```text
nodeUpdateContent=17.49ms/168
nodeUpdateContent=47.24ms/1051
```

最终改为：

```java
updateFully(UpdateCause.ZOOM)
```

效果：

- 保留 shape、edge、cloud、CSS、font、border、attribute、details、note 等布局相关刷新。
- 因为 `UpdateCause.ZOOM.updatesContent()` 为 false，不更新 main text 和 icons。
- 修复部分非自由节点布局错乱，同时避免回到完整内容更新的卡顿。

## details 和 note 修正

后续测试发现 details view 在自由节点和非自由节点缩放时仍可能错位。

原因：

- 自由节点轻量路径原本只 invalidate image viewer。
- details/note 的 min/max width 会受 zoom 后节点宽度影响。
- 如果 details/note 不 revalidate，位置和大小可能滞后于节点布局。

修正：

- 在自由节点 `updateForZoom()` 中新增 `updateViewersForZoom()`。
- 缩放时调用：

```java
nodeViewFactory.updateDetails(this, mainView.getMinimumWidth(), mainView.getMaximumWidth(), UpdateCause.ZOOM);
nodeViewFactory.updateNoteViewer(this, mainView.getMinimumWidth(), mainView.getMaximumWidth(), UpdateCause.ZOOM);
```

因为 cause 是 `ZOOM`：

- 只更新已有 viewer 的 min/max width 和 revalidate。
- 不重新转换 details/note 文本。
- 不触发 map repaint 的完整内容路径。

## 缩放结束后完整 update 的尝试

曾尝试在缩放结束后用延迟 timer 做一次完整 update，避免样式状态遗漏。

结果：

- 缩放过程中较顺。
- 结束后出现明显顿挫，“缩放结束后卡一下”。

因此撤回该方案。当前不再使用缩放结束后的全图完整 update。

当前策略改为：

- 缩放过程中就做必要的轻量布局刷新。
- 对非自由节点保留较完整但不更新内容的 zoom update。
- 对 details/note/image viewer 单独补缩放相关 revalidate。

## 当前效果

已经确认的改善：

- 滚动过程非常丝滑。
- 云框可以通过配置整体关闭或细分控制。
- 简易绘制在任意缩放率下生效。
- 缩放路径不再每帧更新 main text/icons，避免 `nodeUpdateContent` 大量耗时。
- 非自由节点布局比纯轻量路径稳定。
- details/note 在自由节点轻量路径中补了缩放尺寸刷新。

仍需继续观察：

- 大量云框、尤其 arc cloud 的 paint 仍可能是慢帧主因。
- 大量 edge 的绘制在某些视口仍会显著耗时。
- details view 在复杂 HTML、图片、隐藏/展开状态混合时仍需继续测试。
- 非自由节点使用 `updateFully(UpdateCause.ZOOM)` 仍比自由节点轻量路径重，后续可以继续拆分更细的 `updateLayoutForZoom()`。

## 后续可选优化方向

### 1. 非自由节点专用 zoom layout update

当前非自由节点仍会刷新 shape、edge、cloud、CSS、font、border、attribute、details、note。后续可进一步拆出：

```java
updateLayoutForZoom()
```

只保留真正影响树布局的项目。

候选保留项：

- invalidate
- child layout cache reset
- zoom dependent widths
- main view invalidate
- details/note/image viewer revalidate
- 必要的 shape/border/font preferred size 刷新

候选跳过项：

- cloud model/style 重新计算
- edge style 重新计算
- attribute 内容刷新
- background color 读取

需要用日志验证，避免再次引发布局错乱。

### 2. 云框 arc 绘制缓存

云框慢帧中常见热点：

- `cloudCoordinates`
- `cloudDecoration`
- `cloudFill`
- `cloudDraw`
- `cloudText`

可考虑按 zoom、bounds、shape、color、text/image 状态建立短生命周期缓存。需要小心：

- 修改云框类型、颜色、图片后必须清缓存。
- 拖拽期间不能突然隐藏文字/图片。
- 缓存失效策略要覆盖移动、缩放、样式变化。

### 3. edge 可见性裁剪

之前“简易节点跳过 edge”收益明显但语义过粗。更合适的方向是：

- 当父节点和目标节点都在视图外，且 edge 路径不会穿过视口时跳过。
- 只基于可见区域裁剪 edge，而不是基于节点是否简易绘制。

这需要更精确的 bounds 判断。

### 4. details/note 专项验证

建议后续测试覆盖：

- 自由节点 details。
- 非自由节点 details。
- details hidden/expanded 切换。
- HTML details。
- details 内图片或图标。
- 极小缩放和极大缩放。

如果仍出现错位，下一步优先检查 `DetailsView` 的 preferred size 缓存和 HTML label 内部 view 的 invalidate/revalidate。
