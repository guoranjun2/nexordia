# 1.13.3 HTML 图片渲染内存优化

日期: 2026-06-07

## 背景

MainView 和 DetailsView 中的 HTML `<img>` 以前走 Swing 标准 HTML `ImageView` 路径。页面同时显示很多图片时，布局和 preferred span 计算可能同步触发完整图片加载，导致内存峰值过高，严重时出现内存溢出。

优化目标:

- 视口外图片不解码完整像素。
- 低 zoom 按实际绘制尺寸请求缩放图。
- 缩放过程中不清空旧画面，避免闪烁。
- 后台加载完成后可靠触发 repaint。
- 缓存受预算控制，避免重新引入 OOM。

## 实现记录

### 替换 HTML 图片 View

`SynchronousScaledEditorKit` 在创建 HTML view 时识别 `img` 元素，并替换为 `LazyScaledImageView`。这样 MainView 和 DetailsView 里经过 Freeplane HTML renderer 的图片都会走新的 lazy scaled image 路径。

### LazyScaledImageView

`LazyScaledImageView` 负责:

- `getPreferredSpan()` 优先使用 HTML `width` 和 `height`。
- 没有显式尺寸时，只读取图片元数据，不完整解码像素。
- 图片视口外时只画占位，不请求完整图片。
- 图片进入视口后，根据当前 `Graphics2D` transform 和实际 image bounds 计算目标像素尺寸。
- 请求 `HtmlImageCache` 返回缩放后的 `BufferedImage`。
- 如果目标 zoom bucket 正在加载，继续绘制已有同源缓存图，避免缩放闪烁。
- 后台加载完成后 repaint 宿主组件和 viewport，避免图片已加载但屏幕没有刷新的问题。

### HtmlImageCache

`HtmlImageCache` 负责:

- 以规范化 source key、目标宽高 bucket 作为缓存 key。
- 使用 LRU 缓存缩放后的 `BufferedImage`。
- 默认缓存预算为 `32 * 1024 * 1024` 像素。
- 单张目标图最大为 `4 * 1024 * 1024` 像素。
- 目标尺寸按 32 像素 bucket 对齐，减少缩放时重复生成大量相近尺寸图片。
- PNG/JPEG 等 ImageIO 支持的格式使用 subsampling 降低解码内存峰值。
- 后台图片加载线程池为 4 个线程。
- 同一个图片 target 正在 pending 时，会收集所有等待者的 repaint callback，加载完成后一起触发。

### 解决过的问题

1. 内存溢出

   原因是图片在布局或 preferred span 计算阶段过早完整解码。现在只有可见图片才请求目标尺寸像素，并且目标尺寸受 zoom 和缓存预算约束。

2. 缩放闪烁

   原因是新 zoom bucket 未加载完成时回退到占位。现在会查找同源旧缓存图作为 fallback，优先选足够大的最小图，其次选最大的低分辨率图。

3. 拖拽后图片迟迟不出现

   原因是后台加载完成后只 repaint 旧图片 bounds，滚动后该区域可能已经不准确。现在 repaint 宿主组件和 viewport。

4. 多个节点等待同一张图片时只有部分节点刷新

   原因是 pending 图片以前只保留首次请求的 repaint callback。现在 pending 状态会收集所有 repaint callback。

5. 缩放后是否触发新图片渲染

   当前 `paint()` 每次都会按当前 `Graphics2D` transform 计算目标像素尺寸。zoom 改变后，下一次 paint 会请求对应 target bucket。测试已覆盖 transform 影响 target image size。

## Obsidian Canvas 调研

### 已确认事实

Obsidian Canvas 是核心插件，用于在无限画布上组织 notes、images、PDF、video、web pages 等内容。官方文档说明 Canvas 数据保存为 `.canvas` 文件，并使用 JSON Canvas 开放格式:

- https://obsidian.md/help/plugins/canvas
- https://obsidian.md/canvas
- https://jsoncanvas.org/

JSON Canvas 1.0 规范说明 canvas 节点可以是 `text`、`file`、`link`、`group`。图片通常属于 `file` 节点，节点带有固定的 `x`、`y`、`width`、`height`:

- https://jsoncanvas.org/spec/1.0/

Obsidian 本体实现不是开源的，因此无法直接确认其内部图片缓存、虚拟化和缩略图策略。以下分析主要来自 Electron/Chromium 渲染机制、官方文档和社区性能报告。

### 为什么 Obsidian 可能显得更丝滑

1. Electron/Chromium 有成熟的异步渲染管线

   Electron 官方性能文档说明 Chromium 会把重 I/O 和 CPU 密集型工作放到其他线程，避免阻塞 UI 线程:

   - https://www.electronjs.org/docs/latest/tutorial/performance

2. Chromium compositor 会用 GPU 做 transform

   Chromium 的 GPU compositing 文档说明 compositor 会对 layer bitmap 应用 transform，再合成最终画面。主线程忙时，compositor 仍可用已有内容响应滚动和变换:

   - https://www.chromium.org/developers/design-documents/gpu-accelerated-compositing-in-chrome/

3. 缩放时先显示旧 raster，再异步生成更合适分辨率

   Chromium 文档说明 pinch/zoom 时 compositor 会先缩放已有 raster tiles，同时异步按更合适目标分辨率重新 raster，ready 后再替换。快速滚动时也可能先显示低分辨率 tiles:

   - https://www.chromium.org/developers/design-documents/gpu-accelerated-compositing-in-chrome/

   这和当前 Freeplane 的“旧缓存图 fallback，目标 bucket 后台生成”方向一致。

4. 更大的兴趣区域能减少进入/移出视口的抖动

   Obsidian 论坛中有用户通过 CSS 扩大 Canvas wrapper，减少节点在 canvas 边缘被频繁移除/插入，并用 `transition` 和 `will-change` 降低拖动时的 choppiness:

   - https://forum.obsidian.md/t/canvas-sluggish-performance-issue-when-multiple-nodes-enter-exit-the-view/68609/2

   这说明 Obsidian 的丝滑体验并不只来自图片解码，也来自“不要在视口边缘频繁创建/销毁视图”的策略。

### Obsidian 也存在的问题

社区报告显示，Obsidian Canvas 在大画布、图像密集、节点进入/移出视图时仍会卡顿:

- https://forum.obsidian.md/t/canvas-performance-is-irritating/53654
- https://forum.obsidian.md/t/hope-the-official-team-can-optimize-canvas-performance-with-a-summary-of-multiple-test-contents-attached/88023

因此它不是一个可以完全照搬的标准答案。更可靠的参考点是 Chromium compositor 的设计思想:

- 交互过程中优先保证画面连续。
- 复用旧图或低分辨率图。
- 高质量图异步准备。
- ready 后替换。
- 视口附近保留一定 overscan，避免频繁进出导致重建。

## 对 Freeplane 的可借鉴优化

### 已经借鉴并实现

1. 异步图片解码和缩放。
2. 根据 zoom 和实际绘制尺寸请求目标图。
3. 缩放时继续显示旧缓存图。
4. 目标图 ready 后 repaint。
5. 视口外不请求完整图片。

### 建议下一步优化

1. 增加视口预热带和滞后卸载

   当前 `LazyScaledImageView` 使用半个 viewport 作为 margin。可以改成更明确的 hysteresis 策略:

   - 进入距离: viewport 外 1 个屏幕开始预热。
   - 离开距离: viewport 外 2 个屏幕后才停止请求或释放活跃状态。
   - 缓存仍由 LRU 预算控制。

   目的不是让更多图片长期占内存，而是减少拖拽时图片在边缘反复变成 placeholder。

2. 缩放和拖拽期间延迟高质量重绘

   当前 zoom 后下一次 paint 会请求新 bucket。可以进一步加 debounce:

   - 缩放或拖拽进行中，优先显示已有缓存图。
   - 交互停止 80 到 120ms 后，再请求精确目标 bucket。
   - 如果没有任何缓存图，仍请求一个低成本低分辨率图。

   这更接近 Chromium “先缩放 active tiles，再异步 reraster”的模型。

3. 区分快速预览图和最终图

   可以为同一 source 保存两个层级:

   - preview bucket: 小尺寸，快速显示，内存低。
   - exact bucket: 当前 zoom 下更清晰的目标图。

   进入视口时先尝试 preview，稳定后升级 exact。

4. 合并 repaint 请求

   当前 pending callback 已能刷新所有等待者。后续可以进一步做 repaint coalescing:

   - 一个 EDT tick 内把多个图片完成事件合并。
   - 对同一 viewport 只 repaint 一次。

5. 观测和调参

   建议加临时调试计数，不默认输出日志:

   - 当前缓存像素数。
   - pending 图片数量。
   - 命中 exact cache、fallback cache、placeholder 的次数。
   - 图片加载耗时分布。

   这些数据能判断 4 个线程是否真的改善体验，或者只是增加内存峰值。

## 风险

1. 4 个图片加载线程可能提高图片出现速度，但也可能提高瞬时内存峰值。
2. viewport repaint 比局部 bounds repaint 更可靠，但在极端图片密集场景下可能增加重绘面积。
3. 更大的预热带会让体验更平滑，但需要严格依赖 LRU 和像素预算，避免回到 OOM。
4. debounce 高质量重绘会让快速缩放更顺滑，但停止缩放后清晰化会有短暂延迟。

## 结论

Obsidian Canvas 的顺滑感主要值得借鉴的是交互策略，而不是具体实现:

- 交互中保持旧画面连续。
- 后台异步准备更合适的图像。
- 视口附近保留缓冲区。
- 避免节点在视口边界反复创建和销毁。
- ready 后再替换画面。

Freeplane 当前实现已经完成第一轮核心改造。下一步最值得尝试的是“预热带 + 滞后卸载 + 缩放停止后再升级 exact bucket”，这比继续单纯增加线程数更接近 Obsidian/Chromium 的平滑体验来源。
