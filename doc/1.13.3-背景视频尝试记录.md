# 背景视频尝试记录

## 背景

目标是在脑图背景图能力上增加视频类型，已有能力包括静态图片和 GIF。期望视频循环播放，优先作为 map 背景使用，后续可能扩展到节点内嵌视频。

约束和偏好在尝试过程中多次变化：

- 最初希望不用 JavaFX，尝试纯 Java 或原生方案。
- JCodec 纯 Java 解码不依赖系统组件，但播放不流畅，CPU 占用高。
- JavaFX 播放能力更完整，未来节点内嵌视频也可能复用，但在 map 背景里遇到 Swing/JavaFX 混合层级问题。
- 用户明确要求注意内存管理，避免全量读取视频帧。
- 最终本轮决定全部回滚实验代码，只保留复盘文档。

## 已知基线

GIF 方案不是几分钟 GIF 就必然全量撑爆内存。现有 GIF 播放依赖已有图片组件和 ImageIO/动画处理链路，不是应用层显式把所有帧解码后长期缓存成帧列表。

视频和 GIF 的主要差异：

- 视频解码通常依赖硬件或 native pipeline 才能低功耗流畅播放。
- 纯 Java H.264 解码 CPU 成本很高。
- 如果按帧抽图再交给 Swing repaint，性能瓶颈会从视频解码转移到 Swing 重绘和像素拷贝。
- 背景视频不是普通 Swing 图片，层级、透明、重绘、输入穿透都更敏感。

## JCodec 纯 Java 方案

### 实现方向

加入 `org.jcodec:jcodec` 和 `org.jcodec:jcodec-javase`，新增视频 viewer：

- 支持 `.mp4/.m4v/.mov`。
- 后台线程顺序解码。
- Swing EDT 只绘制最近一帧 `BufferedImage`。
- 默认静音、循环、无控制条。
- 不缓存帧列表，只保留当前帧和少量工作区。
- 目标刷新率尝试低频，例如 8 fps。

### 结果

播放不流畅，表现为卡顿、抖动、画面前进一点又后退一点。进一步尝试降低帧率、抽帧、调整节流后仍不理想。

### 结论

JCodec 作为纯 Java 解码方案，内存可控，但 CPU 和流畅度不适合作为高分辨率背景视频方案。它适合验证格式解析或低频缩略图，不适合作为真实背景视频播放后端。

## ffmpeg 抽帧方案

### 实现方向

尝试通过 ffmpeg 抽取帧，再以 Swing 图片组件方式展示。

### 结果

画面可以进一步调清晰度和帧率，但清晰度、流畅度、发热之间互相拉扯。提高帧率和分辨率后发热明显；降低后又失去视频背景意义。

### 结论

抽帧方案本质仍是“视频转图片序列再 repaint”。它避免了部分 Java 解码问题，但引入外部进程、帧传输、像素拷贝和 Swing 重绘压力。作为最终方案不理想。

## JavaFX JFXPanel 方案

### 实现方向

加入 OpenJFX `javafx-base`、`javafx-graphics`、`javafx-media`、`javafx-swing`，通过 `JFXPanel + MediaPlayer + MediaView` 播放本地 MP4：

- 静音。
- 循环播放。
- `MediaPlayer.dispose()` 做释放。
- 视频组件尝试作为背景组件接入现有 `ScalableComponent`。

### 尝试 1：直接作为背景组件绘制

JavaFX 组件不能像普通图片一样走 `paintComponent(Graphics)`。如果把视频当作图片参与 Swing 背景绘制，要么拿不到连续帧，要么又回到截图/重绘/像素拷贝，失去 JavaFX media pipeline 的优势。

结论：JFXPanel 必须作为真实 Swing 组件挂载，不能当成 `paintComponent` 图片源。

### 尝试 2：放进 `JViewport`

为了让视频作为背景层，曾尝试把 `JFXPanel` 加入 `MapViewScrollPane.MapViewPort`。

失败原因：

- `JViewport` 语义上只有一个 `view`。
- Freeplane 代码大量假设 `viewport.getView()` 是 `MapView`。
- 加入 JavaFX 组件后触发：

```text
java.lang.ClassCastException:
JavaFxVideoViewerComponent cannot be cast to MapView
```

结论：不能把视频组件作为 `JViewport` 的 child/view 管理，否则会破坏 `MapViewScrollPane` 的基本假设。

### 尝试 3：放进 `MapView` 内部

尝试把 `JFXPanel` 加进 `MapView`，设置为最底层，并根据 viewport 可见区域调整 bounds。

结果：

- 避免了 `JViewport.getView()` 被破坏。
- 但视频不可见或被 `MapView` 自身绘制/子组件层级吞掉。
- Swing 与 JavaFX 混合绘制中，组件顺序和父组件绘制非常敏感。

结论：直接把视频放进 `MapView` 节点树不可靠，尤其是 `MapView` 有自己的布局、绘制和大量 node view 子组件。

### 尝试 4：`MapViewPane` 外层 `JLayeredPane`

尝试把结构改成：

```text
视频背景层 -> JScrollPane/MapView 透明层 -> overview 浮层
```

具体做法：

- `MapViewPane` 用 `JLayeredPane` 管理层级。
- 视频放在底层。
- `MapViewScrollPane` 和 viewport 设置非 opaque。
- overview 放在更高层。

结果：

- 日志显示 JavaFX 播放器进入 `PLAYING`。
- 日志显示背景组件被定位到正确尺寸，例如 `1849x1051`。
- 但界面仍看不到视频。

推测原因：

- `JFXPanel` 和 Swing 轻量级组件混合在底层时，仍可能被上层 opaque 绘制或 Swing 优化裁剪遮挡。
- macOS 上 JavaFX/Swing 混合、透明父层、JLayeredPane 背景层组合不稳定。
- 即使 `MediaPlayer` 正常播放，底层 JavaFX surface 也未必能通过透明 Swing 层显示出来。

结论：JFXPanel 适合普通嵌入式组件和对话框/节点内可见区域，不适合轻易放到 Swing 复杂组件树的真正背景层。

## JavaFX JDialog 方案

### 现象

单独用 `JDialog` 播放 JavaFX 视频时，发热明显低于一些嵌入背景尝试。视频可以稳定显示，说明 JavaFX MediaPlayer 本身并非完全不可用。

### 差异

JDialog 方案和 map 背景方案的关键差别：

- JDialog 是独立窗口或独立 native surface，JavaFX 可以直接管理播放区域。
- map 背景方案需要嵌入 Freeplane 的 Swing 组件树，并且要在脑图节点后面显示。
- 后者涉及 Swing repaint、透明、层级、输入穿透、窗口激活、最小化、切换 map 生命周期等复杂问题。

### 结论

JavaFX 后续仍可作为“节点内嵌视频”候选，因为节点视频是前景组件，不需要藏到 Swing 背景后面。JavaFX 作为 map 背景则需要谨慎，除非能接受独立窗口覆盖/贴合方案。

## 前台覆盖和透明遮罩方案

### 实现方向

尝试让视频仍在前台，但根据节点位置设置透明区域，让节点露出来。

### 结果

体验很差：

- 页面激活时闪烁。
- 最小化后视频可能单独弹出或残留。
- 切换焦点后循环闪烁。
- 交互和视觉都不稳定。

### 结论

这个方向属于视觉 hack，维护成本和边缘问题太多，不适合继续。

## macOS AVFoundation / AVPlayerLayer 方案

### 实现方向

尝试用 macOS 原生 AVFoundation/AVPlayerLayer，通过 JNA 或 Objective-C runtime 把 `AVPlayerLayer` 插入 AWT/Swing 视图层级。

### 结果

多个阶段结果不同：

- 部分尝试无法显示视频。
- 部分尝试视频显示在前景，遮住脑图节点，无法交互。
- 切换无视频背景 map 后有静态帧残留。
- 再切换回来时背景可能印上前一个脑图视图。
- 插入 AWT view 背后时曾遇到：

```text
native AWT target can not host a CALayer: AWTSurfaceLayers
```

### 结论

AVFoundation 的能耗表现最好，是 macOS 上最接近理想的硬件解码方案。但把 `AVPlayerLayer` 精确放到 Swing/AWT 组件后面并和 Freeplane 生命周期同步，难度很高。这个方向如果继续，需要做成严格 macOS 专用实现，并深入研究 AWTView、CALayer、NSWindow 层级，不适合夹杂在通用 Swing 背景组件里快速试。

## JCEF 方案讨论

### 设想

用 Chromium/JCEF 播放 `<video>`，借助浏览器的视频解码和渲染能力。

### 评估

潜在优势：

- 浏览器视频播放成熟。
- HTML/CSS 控制循环、静音、object-fit 比较简单。
- 后续如果节点内嵌 Web 内容，JCEF 已有生态。

主要风险：

- JCEF 比 JavaFX 更重。
- 与现有插件中的 JCEF 组件、dialog、焦点、窗口层级可能互相影响。
- 作为背景层仍然会遇到 Swing heavyweight/lightweight 混合、透明和遮挡问题。

结论：JCEF 可以作为节点内嵌视频或独立预览候选，但不一定能解决 map 背景层问题。

## 内存结论

所有视频方向都应遵守：

- 不读取整段视频。
- 不缓存所有帧。
- 只保留播放器当前帧/native buffer 或极少量应用侧状态。
- 背景切换、禁用、map remove/关闭时必须停止并释放播放器。

本轮最大问题不是堆内存，而是：

- CPU/GPU 解码和合成成本。
- Swing 重绘。
- JavaFX/Swing/native layer 混合。
- 背景层级和输入穿透。

## 后续建议

### 优先方向 1：节点内嵌视频先行

先不要从 map 背景切入。节点内嵌视频是前景组件，层级问题少很多，适合验证 JavaFX 或 JCEF 的真实播放体验、发热和生命周期。

建议：

- 用 JavaFX `JFXPanel` 做普通节点 viewer。
- 不做透明、不做背景、不做穿透。
- 只支持本地 MP4。
- 明确释放 `MediaPlayer`。
- 观察 10 分钟发热和内存。

如果节点视频稳定，再考虑背景。

### 优先方向 2：macOS 原生专用背景

如果目标是低功耗背景视频，macOS 上 AVFoundation 仍然最有潜力。

但下一次应单独开一个 macOS 专用小分支，先做最小原型：

- 不接 Freeplane 背景配置。
- 不接多 map 生命周期。
- 只在一个简单 Swing frame 中验证 `AVPlayerLayer` 在 AWT/Swing 后层显示。
- 成功后再移植到 MapView。

### 不建议继续的方向

- JCodec 全 Java H.264 播放背景。
- ffmpeg 高频抽帧给 Swing repaint。
- 前台视频加透明遮罩露出节点。
- 在 `JViewport` 中直接加入视频组件。
