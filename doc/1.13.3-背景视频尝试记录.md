# 背景视频尝试记录

## 背景

目标是在脑图背景图能力上增加视频类型，已有能力包括静态图片和 GIF。期望视频循环播放，优先作为 map 背景使用，后续可能扩展到节点内嵌视频。

约束和偏好在尝试过程中多次变化：

- 最初希望不用 JavaFX，尝试纯 Java 或原生方案。
- JCodec 纯 Java 解码不依赖系统组件，但播放不流畅，CPU 占用高。
- JavaFX 播放能力更完整，未来节点内嵌视频也可能复用，但在 map 背景里遇到 Swing/JavaFX 混合层级问题。
- 用户明确要求注意内存管理，避免全量读取视频帧。
- 最终本轮决定全部回滚实验代码，只保留复盘文档。

## 已知基线

GIF 方案不是几分钟 GIF 就必然全量撑爆内存。现有 GIF 播放依赖已有图片组件和 ImageIO/动画处理链路，不是应用层显式把所有帧解码后长期缓存成帧列表。

视频和 GIF 的主要差异：

- 视频解码通常依赖硬件或 native pipeline 才能低功耗流畅播放。
- 纯 Java H.264 解码 CPU 成本很高。
- 如果按帧抽图再交给 Swing repaint，性能瓶颈会从视频解码转移到 Swing 重绘和像素拷贝。
- 背景视频不是普通 Swing 图片，层级、透明、重绘、输入穿透都更敏感。

## JCodec 纯 Java 方案

### 实现方向

加入 `org.jcodec:jcodec` 和 `org.jcodec:jcodec-javase`，新增视频 viewer：

- 支持 `.mp4/.m4v/.mov`。
- 后台线程顺序解码。
- Swing EDT 只绘制最近一帧 `BufferedImage`。
- 默认静音、循环、无控制条。
- 不缓存帧列表，只保留当前帧和少量工作区。
- 目标刷新率尝试低频，例如 8 fps。

### 结果

播放不流畅，表现为卡顿、抖动、画面前进一点又后退一点。进一步尝试降低帧率、抽帧、调整节流后仍不理想。

### 结论

JCodec 作为纯 Java 解码方案，内存可控，但 CPU 和流畅度不适合作为高分辨率背景视频方案。它适合验证格式解析或低频缩略图，不适合作为真实背景视频播放后端。

## ffmpeg 抽帧方案

### 实现方向

尝试通过 ffmpeg 抽取帧，再以 Swing 图片组件方式展示。

### 结果

画面可以进一步调清晰度和帧率，但清晰度、流畅度、发热之间互相拉扯。提高帧率和分辨率后发热明显；降低后又失去视频背景意义。

### 结论

抽帧方案本质仍是“视频转图片序列再 repaint”。它避免了部分 Java 解码问题，但引入外部进程、帧传输、像素拷贝和 Swing 重绘压力。作为最终方案不理想。

## JavaFX JFXPanel 方案

### 实现方向

加入 OpenJFX `javafx-base`、`javafx-graphics`、`javafx-media`、`javafx-swing`，通过 `JFXPanel + MediaPlayer + MediaView` 播放本地 MP4：

- 静音。
- 循环播放。
- `MediaPlayer.dispose()` 做释放。
- 视频组件尝试作为背景组件接入现有 `ScalableComponent`。

### 尝试 1：直接作为背景组件绘制

JavaFX 组件不能像普通图片一样走 `paintComponent(Graphics)`。如果把视频当作图片参与 Swing 背景绘制，要么拿不到连续帧，要么又回到截图/重绘/像素拷贝，失去 JavaFX media pipeline 的优势。

结论：JFXPanel 必须作为真实 Swing 组件挂载，不能当成 `paintComponent` 图片源。

### 尝试 2：放进 `JViewport`

为了让视频作为背景层，曾尝试把 `JFXPanel` 加入 `MapViewScrollPane.MapViewPort`。

失败原因：

- `JViewport` 语义上只有一个 `view`。
- Freeplane 代码大量假设 `viewport.getView()` 是 `MapView`。
- 加入 JavaFX 组件后触发：

```text
java.lang.ClassCastException:
JavaFxVideoViewerComponent cannot be cast to MapView
```

结论：不能把视频组件作为 `JViewport` 的 child/view 管理，否则会破坏 `MapViewScrollPane` 的基本假设。

### 尝试 3：放进 `MapView` 内部

尝试把 `JFXPanel` 加进 `MapView`，设置为最底层，并根据 viewport 可见区域调整 bounds。

结果：

- 避免了 `JViewport.getView()` 被破坏。
- 但视频不可见或被 `MapView` 自身绘制/子组件层级吞掉。
- Swing 与 JavaFX 混合绘制中，组件顺序和父组件绘制非常敏感。

结论：直接把视频放进 `MapView` 节点树不可靠，尤其是 `MapView` 有自己的布局、绘制和大量 node view 子组件。

### 尝试 4：`MapViewPane` 外层 `JLayeredPane`

尝试把结构改成：

```text
视频背景层 -> JScrollPane/MapView 透明层 -> overview 浮层
```

具体做法：

- `MapViewPane` 用 `JLayeredPane` 管理层级。
- 视频放在底层。
- `MapViewScrollPane` 和 viewport 设置非 opaque。
- overview 放在更高层。

结果：

- 日志显示 JavaFX 播放器进入 `PLAYING`。
- 日志显示背景组件被定位到正确尺寸，例如 `1849x1051`。
- 但界面仍看不到视频。

推测原因：

- `JFXPanel` 和 Swing 轻量级组件混合在底层时，仍可能被上层 opaque 绘制或 Swing 优化裁剪遮挡。
- macOS 上 JavaFX/Swing 混合、透明父层、JLayeredPane 背景层组合不稳定。
- 即使 `MediaPlayer` 正常播放，底层 JavaFX surface 也未必能通过透明 Swing 层显示出来。

结论：JFXPanel 适合普通嵌入式组件和对话框/节点内可见区域，不适合轻易放到 Swing 复杂组件树的真正背景层。

## JavaFX JDialog 方案

### 现象

单独用 `JDialog` 播放 JavaFX 视频时，发热明显低于一些嵌入背景尝试。视频可以稳定显示，说明 JavaFX MediaPlayer 本身并非完全不可用。

### 差异

JDialog 方案和 map 背景方案的关键差别：

- JDialog 是独立窗口或独立 native surface，JavaFX 可以直接管理播放区域。
- map 背景方案需要嵌入 Freeplane 的 Swing 组件树，并且要在脑图节点后面显示。
- 后者涉及 Swing repaint、透明、层级、输入穿透、窗口激活、最小化、切换 map 生命周期等复杂问题。

### 结论

JavaFX 后续仍可作为“节点内嵌视频”候选，因为节点视频是前景组件，不需要藏到 Swing 背景后面。JavaFX 作为 map 背景则需要谨慎，除非能接受独立窗口覆盖/贴合方案。

## 前台覆盖和透明遮罩方案

### 实现方向

尝试让视频仍在前台，但根据节点位置设置透明区域，让节点露出来。

### 结果

体验很差：

- 页面激活时闪烁。
- 最小化后视频可能单独弹出或残留。
- 切换焦点后循环闪烁。
- 交互和视觉都不稳定。

### 结论

这个方向属于视觉 hack，维护成本和边缘问题太多，不适合继续。

## macOS AVFoundation / AVPlayerLayer 方案

### 实现方向

尝试用 macOS 原生 AVFoundation/AVPlayerLayer，通过 JNA 或 Objective-C runtime 把 `AVPlayerLayer` 插入 AWT/Swing 视图层级。

### 结果

多个阶段结果不同：

- 部分尝试无法显示视频。
- 部分尝试视频显示在前景，遮住脑图节点，无法交互。
- 切换无视频背景 map 后有静态帧残留。
- 再切换回来时背景可能印上前一个脑图视图。
- 插入 AWT view 背后时曾遇到：

```text
native AWT target can not host a CALayer: AWTSurfaceLayers
```

### 结论

AVFoundation 的能耗表现最好，是 macOS 上最接近理想的硬件解码方案。但把 `AVPlayerLayer` 精确放到 Swing/AWT 组件后面并和 Freeplane 生命周期同步，难度很高。这个方向如果继续，需要做成严格 macOS 专用实现，并深入研究 AWTView、CALayer、NSWindow 层级，不适合夹杂在通用 Swing 背景组件里快速试。

## JCEF 方案讨论

### 设想

用 Chromium/JCEF 播放 `<video>`，借助浏览器的视频解码和渲染能力。

### 评估

潜在优势：

- 浏览器视频播放成熟。
- HTML/CSS 控制循环、静音、object-fit 比较简单。
- 后续如果节点内嵌 Web 内容，JCEF 已有生态。

主要风险：

- JCEF 比 JavaFX 更重。
- 与现有插件中的 JCEF 组件、dialog、焦点、窗口层级可能互相影响。
- 作为背景层仍然会遇到 Swing heavyweight/lightweight 混合、透明和遮挡问题。

结论：JCEF 可以作为节点内嵌视频或独立预览候选，但不一定能解决 map 背景层问题。

## 内存结论

所有视频方向都应遵守：

- 不读取整段视频。
- 不缓存所有帧。
- 只保留播放器当前帧/native buffer 或极少量应用侧状态。
- 背景切换、禁用、map remove/关闭时必须停止并释放播放器。

本轮最大问题不是堆内存，而是：

- CPU/GPU 解码和合成成本。
- Swing 重绘。
- JavaFX/Swing/native layer 混合。
- 背景层级和输入穿透。

## 2026-06-21 JavaFX 背景层重做

### 目标

重新实现视频背景，并且针对 JavaFX 长时间播放后发热的问题做更深入的生命周期优化。重点不是把 JavaFX 视频重新塞进 `paintComponent(Graphics)`，而是继续沿用上次成功的底座方案：

```text
JavaFX 视频组件 -> 透明 MapViewScrollPane/MapView -> overview 浮层
```

### 实现

本轮新增和修改：

- `freeplane/build.gradle`
  - 引入 OpenJFX `javafx-base`、`javafx-graphics`、`javafx-media`、`javafx-swing`。
  - 根据构建机自动选择平台 classifier，例如当前 Apple Silicon 为 `mac-aarch64`。
- `MapBackgroundVideo`
  - 只把 `.mp4`、`.m4v`、`.mov` 识别为视频背景。
  - 支持 `file`、`http`、`https` scheme。
  - 避免视频 URI 再进入静态图片 viewer 链路。
- `ViewerController`
  - 背景选择器额外接受 `.mp4`、`.m4v`、`.mov`。
  - 图片预览组件遇到视频文件时不再尝试用图片 viewer 渲染。
- `MapView`
  - 增加 `setExternalBackgroundPainted(boolean)`。
  - 视频背景可见时，`MapView.paintComponent` 不再填充普通背景色，避免把底层视频盖住。
  - 打印和 overview 仍走普通背景绘制，不把视频帧纳入导出。
- `MapViewPane`
  - 增加底层 `backgroundVideoPanel`。
  - `MapViewScrollPane` 保持为 viewport 的真实 view，避免破坏 `viewport.getView() == MapView` 的既有假设。
  - overview 继续在最上层。
- `JavaFxMapBackgroundVideo`
  - 使用 `JFXPanel + MediaPlayer + MediaView`。
  - 静音、无限循环。
  - `MediaView.setSmooth(false)`，降低缩放滤镜成本。
  - JavaFX scene 和 `JFXPanel` 使用不透明黑色底，避免透明 JavaFX surface 带来的额外合成。

### 发热优化点

本轮相比早期 JavaFX 尝试，主要优化在播放器生命周期：

- 只在 `MapViewPane.isShowing()` 为 true 时创建播放器。
- 切换 tab、关闭 map、组件 remove 时立即 `stop()`、`dispose()`。
- JavaFX `Platform.setImplicitExit(false)`，避免 JavaFX toolkit 在反复创建/销毁时异常退出。
- 视频未 ready 前不把 map 底座切透明，避免显示空黑层。
- 同一个 URI 不重复创建播放器。
- JavaFX 初始化失败或媒体错误时清理视频层，并恢复普通背景绘制。
- dispose 发生在 `MediaPlayer.onReady` 之前也能释放播放器，避免早期异步初始化泄漏。

这些优化解决的是“后台仍然解码”“切换 map 后播放器残留”“初始化失败后透明底座残留”这类问题。只要视频在当前窗口中持续可见播放，解码和合成本身仍然会消耗 CPU/GPU，这是视频背景功能无法完全消除的成本。

### 验证

已执行：

```bash
gradle :freeplane:compileJava
gradle :freeplane:test --tests org.freeplane.view.swing.map.MapBackgroundVideoTest
gradle :freeplane:jar :freeplane:copyOSGiJars :freeplane:copyOSGiManifest
git diff --check
```

结果：

- 编译成功。
- URI 识别单元测试通过。
- `BIN/core/org.freeplane.core/lib/` 中已复制 JavaFX jar。
- `BIN/core/org.freeplane.core/lib/javafx-swing-21.0.5-mac-aarch64.jar` 中包含 `javafx/embed/swing/JFXPanel.class`。
- 生成的 `MANIFEST.MF` 中包含 JavaFX jar 的 `Bundle-ClassPath` 条目。

### 剩余风险

- 当前 Gradle 方案按构建机选择 JavaFX 平台 classifier。在 Windows/Linux 机器上构建时会选择对应平台；如果在 macOS 上交叉产出 Windows/Linux 包，需要额外处理目标平台依赖。
- `JFXPanel` 仍是 Swing/JavaFX 混合组件，不等价于 macOS 原生 AVPlayerLayer。能耗通常优于纯 Java 解码和 Swing 抽帧，但长时间全屏播放仍会发热。
- 视频背景不会进入 overview 缩略图、打印、导出结果。
- 视频默认按 viewport 拉伸填满，极端宽高比素材可能变形。后续可考虑做成 cover/crop，但那会增加一层尺寸计算。
- 目前未增加用户可见开关来控制“窗口失焦时暂停”。如果实际长测仍明显发热，可以继续加窗口激活/最小化生命周期，让 Freeplane 不在后台持续播放。

### 2026-06-21 JavaFX 运行态性能采样

在默认 JavaFX 背景视频路径下，使用 `jcmd`、`jstat` 和 30 秒 JFR 对正在播放视频背景的 Freeplane 进程采样。

运行现场：

- Freeplane 进程 CPU 长时间在 160% 到 200% 左右。
- RSS 从约 1.6GB 继续上升到 3GB 以上。
- Java heap 从约 734MB 上升到 2GB 以上。
- G1 GC 暂停总量很小，30 秒 JFR 中 GC pause 总计约 156ms；发热主因不是 GC。
- full GC 后 live histogram 仍显示 `float[]` 约 1.57GB，说明不只是短生命周期帧垃圾。

关键线程：

- `QuantumRenderer-0`
  - 栈上出现 `com.sun.javafx.media.PrismMediaFrameHandler.updateTexture/getTexture`。
  - 后续栈上出现 `com.sun.prism.es2.GLContext.nReadPixelsInt` 和 `UploadingPainter.run`。
  - 说明 JavaFX 在 `JFXPanel` 嵌入模式下需要把渲染结果从 Prism/GL 渲染目标读回，交给 Swing 合成。
- `Java2D Queue Flusher`
  - 栈上出现 `sun.java2d.metal.MTLRenderQueue.flushBuffer`。
  - 说明 Swing/Java2D Metal 管线也在持续 flush。
- `AWT-EventQueue-0`
  - 线程 CPU 已累计很高，栈上出现 `MapView.paint`、`MapView.paintChildren`、`BezierEdgeView.draw`、`CloudView`、`MainView.paintComponent`。
  - JFR allocation-by-thread 显示 99% 以上 Java 对象分配来自 EDT。

JFR 热点：

- Java 热点第一名是 `sun.java2d.marlin.Renderer._endRendering`。
- native 热点包括：
  - `sun.java2d.metal.MTLRenderQueue.flushBuffer`
  - `com.sun.prism.es2.GLContext.nReadPixelsInt`
  - `com.sun.prism.es2.GLContext.nTexSubImage2D0`
- allocation-by-site 中大量分配来自：
  - `sun.java2d.SunGraphics2D.clone`
  - `sun.java2d.SunGraphics2D.transformShape`
  - `java.awt.geom.Path2D$Float.<init>`
  - `BezierEdgeView.update`
  - `MainView.paintComponent`

机制判断：

当前 JavaFX 背景视频不是独立原生视频层。它是 `JFXPanel`，每一帧都会触发 Swing repaint。为了让视频在下面、脑图在上面，`MapViewPane` 使用了重叠 Swing 组件，并且 `isOptimizedDrawingEnabled()` 返回 false。视频背景组件和透明的 `MapViewScrollPane/MapView` 大面积重叠后，`JFXPanel` 的每帧 dirty region 会迫使 Swing 重新绘制上方的脑图，否则透明叠加顺序不正确。

所以当前发热和内存上涨不是单点问题，而是三段叠加：

1. JavaFX MediaView 播放 60fps 视频，持续解码并上传视频纹理。
2. JFXPanel 嵌入 Swing，需要 `readPixels`，把 JavaFX 渲染结果读回 Swing 可用的图像路径。
3. Swing 重叠透明层让 MapView 跟着视频帧率反复重画，触发 Freeplane 节点、边、云、文字布局和 Java2D Marlin 抗锯齿渲染。

直接后果：

- 即使视频解码本身不是最大瓶颈，视频帧也会把整个脑图绘制路径变成高频任务。
- 大图、云、曲线边、HTML 节点越多，CPU 和分配压力越明显。
- 60fps/高分辨率视频会线性放大这个问题。

优化方向优先级：

1. 停止让视频帧驱动实时 MapView 全量重画。可尝试在视频播放期间用透明截图层缓存脑图，只有缩放、滚动、选择、编辑、map 改变时刷新截图。
2. 缓存高频几何对象，例如 `BezierEdgeView.update()` 产生的 `GeneralPath`，避免每次 repaint 为每条边创建新路径。
3. 给视频背景加低功耗策略：窗口失焦、最小化、非当前 map、长时间无交互时暂停。
4. 降低视频素材成本：优先使用 720p/1080p 30fps，避免 60fps 高码率视频作为背景。
5. 若继续追求低功耗，仍应回到原生视频层或独立窗口方案；`JFXPanel` 嵌入 Swing 的架构上限已经暴露。

### 2026-06-21 JavaFX 前景覆盖实验

为了验证发热到底来自 JavaFX 自身，还是来自“透明脑图叠在视频上方导致 MapView 被视频帧拖着重画”，新增临时开关：

```text
-Dorg.freeplane.backgroundVideo.frontTest=true
```

该开关只用于实验：

- 视频层放到 MapView 上方。
- `MapViewPane`、`MapViewScrollPane`、`JViewport` 和 `MapView` 保持普通不透明绘制。
- 不调用 `mapView.setExternalBackgroundPainted(true)`。
- 结果是视频会盖住脑图，不能作为真实功能，只用于隔离性能因素。

采样结果：

- CPU 稳定在约 90% 到 100%，仍然偏高，但低于透明背景模式的 160% 到 200%。
- heap 没有继续冲到 2GB 到 3GB；采样时约 600MB 到 870MB。
- live histogram 中 `float[]` 从透明背景模式的约 1.57GB 降到约 198MB。
- 20 秒 JFR 中 GC pause 只有约 5ms。
- JFR Java execution samples 从透明背景模式的 691 个降到 4 个。
- EDT 线程快照处于 WAITING，不再卡在 `MapView.paint`、`BezierEdgeView.draw` 或 `CloudView`。
- JavaFX 仍然在 `GLContext.nReadPixelsInt` 和 `GLContext.nTexSubImage2D0`，说明 `JFXPanel` 的读回/上传成本仍在。

结论：

- 透明叠层确实是此前内存暴涨和 MapView 高负载的主要放大器。
- JavaFX/JFXPanel 自身仍然需要约一个核心左右的持续成本，主要来自嵌入 Swing 后的视频帧读回和纹理上传。
- 若要保留“脑图在视频上方”的视觉效果，下一步应尝试脑图截图缓存层，而不是让真实 `MapView` 以视频帧率重画。

### 2026-06-21 JavaFX 脑图截图缓存实验

为了避免视频帧驱动真实 `MapView` 反复重画，尝试了截图缓存层：

- 视频仍由 `JFXPanel + MediaView` 播放。
- 真实 `MapView` 在视频播放时保持不透明或不参与透明重绘。
- 只在启动、布局变化等时把当前脑图绘制到一张 `BufferedImage`。
- 第一版用 Swing 透明面板把截图盖在视频上。
- 第二版把截图转换成 JavaFX `ImageView`，直接叠在同一个 JavaFX scene 中，避免 Swing 透明截图层每帧参与合成。

结果：

- 内存暴涨问题明显缓和，没有再出现透明背景模式下 `float[]` live size 超过 1GB 的情况。
- 但 CPU 仍可回到 170% 左右。
- 线程栈显示瓶颈不再是 `MapView.paint`，而是：

```text
AWT-EventQueue-0
  javafx.embed.swing.JFXPanel.paintComponent
  com.sun.javafx.tk.quantum.EmbeddedScene.getPixels

QuantumRenderer-0
  com.sun.javafx.tk.quantum.UploadingPainter.run
  com.sun.prism.es2.GLContext.nReadPixelsInt
```

JFR 也显示 native 热点集中在：

- `com.sun.prism.es2.GLContext.nReadPixelsInt`
- `sun.java2d.metal.MTLRenderQueue.flushBuffer`
- `com.sun.prism.es2.GLContext.nTexSubImage2D0`

结论：

- 截图缓存能解决“视频帧拖着脑图全量重画”的问题。
- 但只要还使用 `JFXPanel`，JavaFX embedded scene 每帧仍要把渲染结果读回给 Swing。
- 这个像素读回/拷贝路径足以让菜单交互变卡，并继续造成明显发热。
- 所以截图缓存不是最终答案，只能证明 MapView 重绘是一个放大器；真正的底层瓶颈仍是 `JFXPanel` 嵌入模型。

### 2026-06-21 独立 JavaFX Stage 实验

为了验证 JavaFX 媒体本身是否可接受，新增临时开关：

```text
-Dorg.freeplane.backgroundVideo.stageTest=true
```

该开关只用于实验：

- 不创建 `JFXPanel`。
- JavaFX 自己创建无边框 `Stage`。
- `MediaView` 在独立 JavaFX 窗口里播放。
- Stage 根据 Freeplane map viewport 的屏幕坐标同步位置和尺寸。
- Stage 当前放在前景并 `alwaysOnTop`，会盖住脑图，不是最终交互方案。

采样结果：

- CPU 稳定约 83% 到 85%。
- RSS 约 1.3GB 到 1.47GB，低于同场景 `JFXPanel` 前景测试约 1.7GB。
- EDT 线程不再卡在 `JFXPanel.paintComponent`，线程快照中 EDT 回到等待事件状态。
- JFR native 热点中 `nReadPixelsInt` 消失。
- 主要剩余热点变成正常视频纹理上传：

```text
com.sun.prism.es2.GLContext.nTexSubImage2D0
com.sun.prism.es2.MacGLDrawable.nSwapBuffers
```

结论：

- JavaFX MediaPlayer/MediaView 本身不是最大问题。
- 真正导致高热和菜单卡顿的是 `JFXPanel` 把 JavaFX scene 嵌入 Swing 时的每帧读回/合成。
- 如果继续用 JavaFX，下一步应该围绕“独立窗口播放”设计，而不是继续优化 `JFXPanel`。

下一步实验方向：

- 独立 Stage/Window 放到主窗口后面。
- 主 Swing 窗口只在 map viewport 区域形成透明洞。
- 研究 click-through / 不抢焦点 / 跟随主窗口移动和最小化。
- 如果这条路可见且交互可控，再考虑把 Stage 路径作为跨平台低功耗背景的候选。

## 2026-06-21 方案 3：macOS AVPlayerLayer 实验实现

### 背景

JavaFX 背景视频已经比最早版本好一些，但长时间播放仍会明显发热，并且菜单交互会变卡。继续分析后，主要瓶颈不只是视频解码，而是 `JFXPanel` 嵌入 Swing 的显示路径：JavaFX embedded scene 会把画面像素同步给 Swing，再由 Swing/EDT 绘制。视频持续刷新时，这相当于持续触发大面积像素拷贝和 Swing 重绘，菜单、弹窗和脑图交互都会被同一个 UI 管线影响。

所以方案 3 的目标是绕开 `JFXPanel` 这条路径：在 macOS 上尝试直接把 AVFoundation 的 `AVPlayerLayer` 插到 AWT 窗口对应的原生 NSView layer 底部，让系统媒体框架负责视频解码和合成。

### 本次实现

- 新增 `MapBackgroundVideoPlayer` 抽象，把背景视频播放器从 `MapViewPane` 中拆出来。
- 新增 `MapBackgroundVideoPlayers` 工厂：macOS 优先反射创建 `org.freeplane.plugin.macos.MacNativeMapBackgroundVideo`，失败时回退 JavaFX。
- 保留 `JavaFxMapBackgroundVideo` 作为跨平台默认实现。
- 在 `freeplane_mac` 中新增 `MacNativeMapBackgroundVideo`，通过 JNA 调 Objective-C runtime、AVFoundation 和 CoreAnimation。
- `freeplane` 核心只依赖反射字符串，不直接 import macOS 类，也不直接依赖 AppKit/AVFoundation API。
- `copyOSGiJars` 现在也复制 `maclib`，否则 JNA 只在 manifest 中出现或只在本地缓存中出现，运行时会找不到类。

### 架构影响

这个方案没有改 MapView 的主体绘制模型：脑图仍然在 Swing 里画，背景视频仍然由 `MapViewPane` 统一管理生命周期。变化主要集中在播放器实现层：

- 非 macOS 仍然走 JavaFX。
- macOS 默认走 JavaFX。
- 可用 `-Dorg.freeplane.backgroundVideo.nativeMac=true` 显式启用 macOS 原生 AVPlayerLayer 路径，用于和 JavaFX 做 A/B 对比。
- 原生安装失败会自动换 JavaFX。
- JavaFX 播放失败才清理背景视频。
- 切换 map、隐藏窗口、关闭组件时会 dispose 当前播放器。

这意味着方案 3 对现有架构的侵入相对可控。真正的不确定性集中在 macOS 原生 layer 和 AWT/Swing 绘制层的组合关系上。

### 预期收益

- 避免 `JFXPanel` 的逐帧 JavaFX -> Swing 像素拷贝。
- 减少 EDT 被视频刷新拖慢的概率。
- 让视频解码和合成更多交给 AVFoundation/CoreAnimation。
- 如果 layer 能稳定位于 Swing 背景下方，菜单卡顿和发热应该比 JavaFX 路径明显改善。

### 主要风险

- AWT 顶层窗口或 Swing viewport 如果仍然拥有不透明原生 surface，AVPlayerLayer 可能被挡住，表现为“没有背景”。
- `AVPlayerLayer` 插入的是 AWT NSView 的 layer tree，和 Swing repaint 的交互依赖 JDK macOS 实现细节。
- 需要 JDK 暴露 `sun.awt`、`sun.lwawt`、`sun.lwawt.macosx`。当前启动脚本已有相关 `--add-exports`/`--add-opens`，但这是非公共 API。
- JNA/Objective-C selector 调用缺少编译期类型保护，问题多发生在运行时。
- 这条路径只能优化 macOS。Windows/Linux 仍然使用 JavaFX，除非后续分别接各自原生媒体层。

### 当前判断

方案 3 是正确的性能方向，但它不是一个纯 Java 层面的确定性优化。接下来要用真实 Freeplane 窗口验证两个问题：

- 是否能看见背景视频。
- 菜单交互和长时间播放发热是否明显好于 JavaFX。

如果“看不见”，下一步要研究的是 AWT/Swing 原生 surface 的透明度和 layer 层级；如果“看得见但仍发热”，再继续做窗口失焦/最小化暂停、降低视频分辨率、限制背景刷新区域等次级优化。

### 2026-06-21 运行反馈

第一次重启后仍然明显卡顿。线程栈显示当前实际路径仍然是 JavaFX：

```text
JavaFX Application Thread
QuantumRenderer-0
JFXMedia Player EventQueueThread
javafx.embed.swing.JFXPanel.paintComponent
SunGraphics2D.drawImage
```

日志显示 macOS 原生路径失败后降级到了 JavaFX。先后遇到两个反射访问问题：

- `IllegalAccessException: ... java.awt.Component$1.getPeer(...)`
- `InaccessibleObjectException: module java.desktop does not "opens java.awt" ...`

处理方式：

- 不再通过 `componentAccessor.getClass()` 反射调用匿名实现类。
- 改为通过 `sun.awt.AWTAccessor$ComponentAccessor` 接口取得 `getPeer(Component)` 方法。
- 原生路径 ready 后增加日志：`macOS native background video ready: ...`。

因此下一次验证时，必须先确认日志中出现 `macOS native background video ready`，并且线程栈中不再出现 `JFXPanel.paintComponent` 作为视频绘制路径。否则测试到的仍然是 JavaFX fallback，不代表方案 3 的真实效果。

第二次重启后出现 `macOS native background video ready`，说明原生路径已启动，但随后 UI 卡死。线程栈显示死锁：

- EDT 在 `MapViewPane.updateBackgroundVideoVisibility()` 调用 `backgroundVideoPanel.setVisible(true)`，持有 `AWTTreeLock`。
- 该过程触发 `HierarchyListener`，进入 `MacNativeMapBackgroundVideo.syncBounds()`。
- `syncBounds()` 同步等待 AppKit。
- AppKit 线程执行 `SwingUtilities.convertRectangle()`，又尝试获取 `AWTTreeLock`。

处理方式：

- bounds 计算改在 EDT 上延迟执行，避开 hierarchy event 正持有 `AWTTreeLock` 的阶段。
- AppKit 线程只接收已经算好的 `CGRect` 并设置 `AVPlayerLayer` frame。
- `play()`、`pause()`、`dispose()`、bounds 同步改为异步投递到 AppKit，避免 EDT 同步等待 AppKit。

同一轮还发现视频盖住脑图。原因是 `AVPlayerLayer` 被作为 AWT NSView backing layer 的 sublayer 插入；CALayer 的 sublayer 总是在 parent contents 上方绘制，所以 `insertSublayer:atIndex:0` 仍会盖住 Swing 内容。

处理方式：

- 改为获取 AWT NSView 的 `superview`。
- 将 `AVPlayerLayer` 插到 superview 的 layer 中，并放在 AWT layer 下面：`insertSublayer:below:`。
- 这样才有机会成为真正背景层。若后续完全看不见视频，说明 AWT layer 本身仍是不透明 surface，需要继续研究窗口/NSView 透明度。

第三次重启后原生路径 ready、不卡，但视频不可见。组件层级分析：

```text
MapViewPane
  backgroundVideoPanel
  mapOverviewPanel
  mapViewScrollPane
    MapViewPort / JViewport
      MapView
        NodeView...
```

Swing 侧已有透明处理：

- `mapViewScrollPane.setOpaque(false)`
- `viewport.setOpaque(false)`
- `mapView.setExternalBackgroundPainted(true)`，让 `MapView.paintComponent()` 不再填背景

但 native 视频已经在整个 AWT layer 下面，不是普通 Swing 子组件。若上方任意 Swing 容器或 AWT backing layer 仍填充不透明背景，视频都会被挡住。

处理方式：

- `MapViewPane` 在视频可见时也设为 `opaque=false`。
- macOS native 路径中调用 `CPlatformView.setWindowLayerOpaque(false)`。
- AWT layer 和 superview layer 调 `setOpaque:NO`，并把 `backgroundColor` 设为 `NSColor.clearColor.CGColor`。

第四次验证后视频仍不可见，且日志仍显示 native ready。进一步判断：只把 `CPlatformView` 的 window layer 设为透明仍不够，顶层 `CPlatformWindow`/NSWindow 仍可能按不透明窗口合成。

处理方式：

- 安装 native 视频时调用 `CPlatformWindow.setOpaque(false)`。
- dispose 时恢复 `CPlatformView.setWindowLayerOpaque(true)` 和 `CPlatformWindow.setOpaque(true)`。

如果这一步仍不可见，说明 AWT/Java2D Metal surface 很可能不支持这种“视频 layer 在 AWT layer 下方、Swing 透明穿透”的组合。下一步应停止继续补透明属性，转向独立 NSWindow/JDialog 与主窗口同步位置。

第五次验证后视频仍不可见，日志仍显示 `macOS native background video ready`，且没有降级到 JavaFX。这个结果基本排除了“播放器没启动”和“Swing 子组件 opaque 没清干净”两个方向：原生视频层能创建成功，之前放到 AWT layer 上方时也能盖住脑图；但一旦放到 AWT layer 下方，Swing/AWT 的原生 backing layer 仍然按不透明 surface 合成，透明 Swing 组件无法真正穿透到下面的 sibling CALayer。

第六次验证改用独立 `NSWindow`，但不再放到主窗口后方，而是作为主窗口的 child window 放到上方：

- `AVFoundation` / `AVQueuePlayer` / `AVPlayerLooper` 继续负责播放和循环。
- 视频窗口设置 `setIgnoresMouseEvents:YES`，鼠标事件穿透到 Freeplane。
- 视频窗口设置 `setAlphaValue:`，用透明度把前景视频降成背景视觉。
- `MapBackgroundVideoPlayer.isPaintedInFront()` 让 Swing 侧知道这个播放器不需要 host window 透明。
- macOS 默认启用 native 路径；显式设置 `-Dorg.freeplane.backgroundVideo.nativeMac=false` 才回退 JavaFX。

验证结果：

- `-Dorg.freeplane.backgroundVideo.nativeMac.opacity=0.8` 时，截图确认星空视频明显显示在脑图区域上方。
- 默认 `opacity=0.35` 时，截图 `/tmp/freeplane-native-front-default.png` 确认视频仍然可见，脑图文字仍可读。
- 连续 CPU 采样约 2% 到 3%，明显低于 JavaFX/JFXPanel 路径。
- 用户目录加载正常，日志出现 `/Users/peter/.freeplane/1.12.x/scripts/init/Starter.groovy`。

当前结论：不要继续在 Freeplane 主窗口里给 AWT layer 补透明属性。问题已经从 Swing 组件透明变成 JDK macOS AWT/Metal surface 与 CoreAnimation sibling layer 的合成边界。当前可落地方案是 macOS 原生 AVFoundation 独立前置穿透窗口，用透明度控制视觉干扰；它不是严格意义上的后置背景，但满足“可见、低发热、鼠标交互不被视频抢走”的核心要求。

- `JavaFxStageMapBackgroundVideo` 实验类已删除，最终 `freeplaneviewer.jar` 中不再包含这条闪烁路线。

### 透明度修改后的交互问题

问题现象：修改前景视频透明度后，回到脑图区域需要额外按 Enter 才能正常交互。

原因判断：`background_video_foreground_opacity` 变化时，`MapViewPane` 原先会 `clearBackgroundVideo()` 再 `updateBackgroundVideo()`。macOS native 路径下这会销毁并重新创建前置透明 child `NSWindow`，触发窗口 ordering、焦点和事件状态变化。视频窗口本身虽然 `setIgnoresMouseEvents:YES`，但重建窗口仍然可能扰动当前 Swing 焦点状态。

修复方式：

- `MapBackgroundVideoPlayer` 增加 `updateForegroundOpacity()`。
- `MapViewPane` 收到透明度属性变化时不再重建播放器，只调用当前播放器更新透明度。
- `MacNativeMapBackgroundVideo.updateForegroundOpacity()` 在 AppKit 线程对已有 `videoWindow` 调用 `setAlphaValue:`。
- JavaFX/JFXPanel 路径默认不处理这个属性，因为它不是前置透明窗口，不需要通过窗口 alpha 做背景化。

当前结论：

- `Stage` 路径不适合作为 Freeplane 背景视频实现，会带来焦点闪烁风险。
- macOS 上继续以 AVFoundation native child window 作为主线；它目前是唯一同时满足可见、低 CPU、鼠标穿透、窗口同步的路线。
- 如果之后继续研究 JavaFX 低发热方案，应改成 AWT 非焦点窗口承载 `JFXPanel`，例如 `JWindow/JDialog + setFocusableWindowState(false)`，而不是 JavaFX `Stage`。
