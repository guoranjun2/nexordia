# 0.1.0 HTML GIF 放大刷新区域修复

日期: 2026-06-09

## 背景

HTML 节点中 GIF 图片在高 zoom 下会出现局部区域没有随整体缩放刷新的问题。视觉表现是静态背景或上一帧已经按节点缩放放大，但 GIF 动画更新区域仍像原始尺寸一样，只刷新画面中的一小块。

这个问题不属于 GIF 解码失败，也不是普通的图像插值质量问题。根因是 Swing 默认 `ImageView` 的 GIF 帧刷新由内部 `ImageObserver` 触发，它记录和 repaint 的区域来自未缩放 view 坐标；Freeplane 的 `ZoomableLabelUI` 在绘制时会缩放整个 `Graphics2D`，因此默认 GIF repaint 区域没有跟着 map zoom 放大。

## 修改文件

生产代码:

- `freeplane/src/main/java/org/freeplane/core/ui/components/html/ScaledGifImageView.java`
- `freeplane/src/main/java/org/freeplane/core/ui/components/html/SynchronousScaledEditorKit.java`

测试代码:

- `freeplane/src/test/java/org/freeplane/core/ui/components/html/SynchronousScaledEditorKitTest.java`

## 实现记录

### 保留默认 GIF 加载机制

GIF 不回到 `LazyScaledImageView`，也不重新引入自定义 GIF 帧缓存。原因是 Swing 默认 HTML `ImageView` 已经能通过 `Toolkit` 和 `ImageObserver` 支持 GIF 动画、加载状态和帧推进。

新的 `ScaledGifImageView` 仍继承 `javax.swing.text.html.ImageView`，继续使用 `getImage()` 取得默认加载出的 GIF `Image`，避免重新实现 GIF 解码和动画生命周期。

### 修正 repaint 区域

`ScaledGifImageView` 在绘制时记录当前 HTML image view 的 allocation，并按当前 map zoom 计算组件坐标里的 repaint bounds。

GIF 帧更新时，自定义 `ImageObserver` 不再使用 JDK `ImageView` 内部未缩放 bounds，而是调用宿主组件的:

```java
component.repaint(0, scaledX, scaledY, scaledWidth, scaledHeight);
```

这样动画帧刷新的脏区和 Freeplane 节点整体缩放后的显示区域一致，避免高 zoom 下只刷新原始小区域。

### 绘制质量

`ScaledGifImageView` 绘制时使用 `Graphics2D` 副本，并设置:

- `RenderingHints.KEY_INTERPOLATION = VALUE_INTERPOLATION_BILINEAR`
- `RenderingHints.KEY_RENDERING = VALUE_RENDER_QUALITY`

这些 hint 只影响当前 GIF 绘制副本，`dispose()` 后不会污染外层 `Graphics2D` 状态。

### EditorKit 分流

`SynchronousScaledEditorKit` 创建 `<img>` view 时继续按格式分流:

- GIF 使用 `ScaledGifImageView`
- 非 GIF 使用 `LazyScaledImageView`

因此静态图片仍保留视口外懒加载、按 zoom 缩放缓存和后台加载优化；GIF 则保留默认动画机制，同时修正 zoom 后的帧刷新区域。

## 性能影响

与原来的默认 GIF 路径相比:

- GIF 解码和动画推进成本基本不变，仍使用 Swing 默认 `ImageView` 加载出的 `Image`。
- 没有新增 GIF 帧缓存，也没有生成缩放后的 `BufferedImage` 序列，内存峰值变化很小。
- 每次绘制多一次 `Graphics.create()`、两个 rendering hint 设置和一次 `dispose()`，开销很小。
- 主要成本变化来自 repaint 面积变大。旧逻辑只刷新未缩放区域，虽然成本低但显示错误；新逻辑按 zoom 刷新完整 GIF 显示区域。高 zoom 下 repaint 像素数大约随 zoom 面积增长，例如 3x zoom 约为 9 倍像素面积。

结论: 这次修复没有增加解码或缓存压力，主要是把原先错误偏小的 repaint 区域扩大到正确尺寸。少量 GIF 的性能接近原路径；大 GIF、高 zoom、多个 GIF 同时播放时，绘制负担会随正确刷新面积增加。

## 失败尝试

第一次尝试只包装默认 `ImageView` 并设置 Java2D 插值 hint。这个方案能改善放大插值质量，但不能解决 GIF 区域没有放大的问题，因为真正的问题在动画帧触发 repaint 时使用了未缩放脏区。

后续根据实际截图重新定位到 repaint bounds，改为自定义 observer 并按 map zoom 放大 repaint 区域。

结果均为 `BUILD SUCCESSFUL`。

- GIF repaint bounds 会按 zoom 转成放大后的组件坐标。
