# 0.1.0 面板折叠状态与节点折叠 handle 扩展

## 背景

本轮改动处理两类交互问题：

- 左右侧面板折叠后，重启或重建窗口时没有保留折叠状态。
- 单独新增节点创建 handler 后，位置、缩放、折叠标记和节点拖拽之间容易相互干扰。

最终方案取消独立的节点创建 handler，改为扩展已有节点折叠 handle：点击仍处理折叠相关动作，拖拽则用于创建连接或创建自由节点。

## 面板折叠状态

`ModernToolWindowPanel` 现在会按面板位置保存折叠状态。左侧和右侧面板折叠后，状态写入用户属性；再次启动或重新创建面板时，会按属性恢复。

这部分只保存侧栏自身的折叠/展开状态，不改变面板内容、停靠位置或已有宽度设置。

## 折叠 handle 的新行为

折叠 handle 现在也会显示在没有子节点的节点上。行为按操作区分：

- 有子节点时点击 handle：沿用原折叠/展开行为。
- 没有子节点时点击 handle：创建普通子节点。
- 从 handle 拖拽到另一个节点：创建从源节点到目标节点的连接线。
- 从 handle 拖拽到空白区域：创建默认子节点，然后把该节点标记为自由节点并移动到拖拽释放位置附近。

拖拽预览线从源节点中心出发，不从 handle 出发。拖到目标节点上时，目标节点有外框提示，预览线使用当前标准连接线形状；落到空白处时只创建自由节点，不额外创建连接线。

## 自由节点创建定位

自由节点初始定位逻辑抽到 `FreeNodePlacement`：

- 根据父节点布局方向判断主轴和副轴。
- 根据落点在父节点主轴中心线的哪一侧设置节点 side。
- 对 `TOP_OR_LEFT` / `BOTTOM_OR_RIGHT` 这类强制子节点方向的布局，保留用户实际拖拽方向对应的视觉落点。
- 新节点完成布局后再做一次居中修正，让节点中心尽量贴近鼠标释放点。

已知风险：首个或高位自由节点的纵向位置仍可能受到 `NodeView` 后续布局扩展影响，表现为比鼠标释放点更高。之前尝试过改鼠标坐标换算，但没有解决根因，已去掉该无效修复。后续如果继续处理，应优先从 `NodeView` 布局后位置和 `LocationModel` 的主轴/副轴语义入手。

## 折叠 handle 滚轮缩放自由子节点距离

折叠 handle 上的滚轮行为做成 `FoldingControlGestureAction` 枚举，便于后续继续扩展同一区域的手势动作。当前包含距离缩放和自由子节点旋转两个动作。

只有鼠标指针位于折叠 handle 范围内，并且当前 map 可编辑时，滚轮才会触发该行为：

- 向上滚动：把直接自由子节点推远。
- 向下滚动：把直接自由子节点拉近。
- 只处理当前节点的直接子节点，并且只处理 `FreeNode`。
- 不再拦截触控板双指张开/合并；触控板缩放仍走原地图缩放逻辑。
- 鼠标位于折叠 handle 上时，`MNodeMouseWheelListener` 会先消费滚轮事件并返回，不再进入地图滚动逻辑。动作是否实际移动节点不影响事件消费。

距离缩放逻辑抽到 `FreeNodeDistanceScaler`。它使用父节点中心和自由子节点中心之间的向量缩放，再写回子节点的 `hGap` 和 `shiftY`。这样连续推远再拉近时，节点更接近原位置，不会只改 x 或只改 y。

## Shift 滚轮旋转自由子节点

`Shift + 滚轮` 用于旋转折叠 handle 所在节点的直接自由子节点。该动作只在鼠标位于折叠 handle 上，并且真实键盘 Shift 处于按下状态时生效；离开 handle 后，滚轮事件继续走原地图滚轮逻辑。

对 handle 所在父节点，查找直接自由子节点，并围绕父节点中心旋转这些自由子节点：

- 滚轮向上：顺时针旋转。
- 滚轮向下：逆时针旋转。
- 每个标准滚轮刻度旋转 5 度。
- 旋转时同时更新 `hGap` 和 `shiftY`，不做单轴偏移。

旋转计算抽到 `FreeNodeRotator`。它和距离缩放一样基于未缩放的 `MainView` 尺寸计算父节点中心、子节点中心，再把旋转后的中心位置转换回子节点偏移。

触控板下不使用 `Ctrl + Shift + 滚轮` 做旋转，因为 macOS/Java 对 `Ctrl + 滚轮` 的事件节奏不如普通滚轮连续，实际体验会出现断续感。旋转也不直接依赖 `MouseWheelEvent.isShiftDown()`，因为触控板横向滚动可能合成 Shift 滚轮事件；代码通过真实键盘 Shift 状态区分旋转和推远/拉近。

触控板慢速 `Shift + 滚动` 的连续性仍可能受系统事件节奏影响。当前保留普通滚动和 `Shift + 滚动` 两种手势，不再引入额外字母键或复杂模式切换。

## 放弃或修正过的方案

- 独立的节点创建 handler 容易和折叠 handle 重合，也难以在所有布局和缩放下稳定命中，因此改回绑定 `MainView` 并复用折叠 handle。
- 拖拽释放到空白处不创建浮动节点对象，而是按默认节点创建流程创建节点，再添加 `FreeNode` 扩展。
- 拖拽到其他节点才创建连接线；拖拽到空白处只创建自由节点。
- 对鼠标释放点做额外换算的尝试没有解决高位自由节点偏移问题，已移除。
- 推远/拉近不按单轴处理，而是按父子节点中心向量同时缩放 x、y。

## 后续风险

折叠 handle 是强 UI 行为，仍需要在多种布局、缩放比例、折叠状态和高 DPI 屏幕上手工验证。尤其是自由节点创建后的最终位置，仍可能被 `NodeView` 布局更新影响。

滚轮缩放当前只处理直接自由子节点。如果以后需要处理所有层级自由后代，应单独定义遍历范围和连接线更新策略，避免一次滚轮影响过大。
