# 0.1.0 HTML image rendering load

## Context

Rendering mind maps with many non-GIF HTML node images caused heavy CPU load, especially while zooming out. The low zoom case exposes many node images at once, causing many scaled image loads and repaint callbacks.

## Root cause

`LazyScaledImageView` scheduled image loads during painting, and `HtmlImageCache` kept every pending target size request. Fast zooming could enqueue obsolete target sizes for the same image source. Each completed load also triggered repaint callbacks independently, and `LazyScaledImageView` repainted the whole host component and viewport.

The issue was most visible when zooming out because many images became visible together. It was less visible when zooming in because fewer images were visible.

## Changes

- Reduced non-GIF HTML image loader concurrency from four threads to two to lower peak decode pressure.
- Kept only the latest pending target size per image source; obsolete queued loads are skipped before decoding, and obsolete completed loads are flushed instead of cached.
- Coalesced pending repaint callbacks so multiple image loads schedule one EDT repaint pass.
- Changed image repaint from whole host and viewport repaint to the last painted image bounds.
- Kept large zoomed images visible through normal viewport intersection checks.
- Grouped requested target sizes into fixed image levels from 16 to 2048 pixels so nearby zoom states share cached images instead of producing many short-lived scaled variants.
- Increased the global scaled-image cache from 32 MP to 64 MP.
- Included local file modification time in image cache keys so edited images do not reuse stale scaled variants from the previous file contents.

GIF images are unchanged and still use the default GIF rendering path.

## Verification

- `gradle :freeplane:test --tests 'org.freeplane.core.ui.components.html.*'`
- `gradle :freeplane:compileJava`

Both commands passed.

## Follow-up

If more optimization is needed, the next step is to make the queue viewport-prioritized, so current-screen images load before preload-margin images.
