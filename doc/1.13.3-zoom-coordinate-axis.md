# 1.13.3 Zoom and Coordinate Axis Notes

## Background

This work focused on improving low zoom behavior and adding a coordinate axis overlay for map inspection.

The first semantic zoom idea was abandoned because Freeplane uses zoom during Swing layout. Freezing layout zoom while allowing visual zoom to keep changing caused the view to feel inconsistent: positions and scroll behavior could stop matching the visible scale. The current implementation keeps the original geometric zoom model and only makes the lower zoom limit configurable.

## Zoom Configuration

`map_view_zoom_shift` controls the zoom ratio applied for each wheel or trackpad magnification step. The default is `0.04`.

`map_view_min_zoom` controls the minimum allowed map view zoom. The default is `0.001`, which means `0.1%`.

Both mouse wheel zoom and macOS trackpad magnification use the same `ZoomCalculator` path, so they share the configured minimum and the existing maximum of `32.0`.

Setting `map_view_min_zoom` to `0` disables the lower clamp.

## Coordinate Axis Overlay

`showCoordinateAxis` enables a map overlay with coordinate axes, grid lines, and labels.

The option is placed under the `coordinate_axis` separator in the Painting preferences tab. Low zoom and cloud painting options remain under `low_zoom_painting`.

The overlay is painted only for normal painting, not for printing or other paint purposes.

The origin is anchored at the root node main view center. This makes coordinates useful for comparing node positions relative to the root.

## Grid Spacing

The grid uses screen-stable spacing:

- Labels are based on unscaled map coordinates.
- Line positions are projected to screen coordinates using the current zoom.
- The grid calculator chooses a map unit step that keeps visible line spacing roughly between `60px` and `160px`.

This avoids the previous behavior where the calculated map step was drawn directly as a screen pixel step. That made labels and visible node positions drift when zoom changed.

Grid lines are drawn with double precision using `Line2D.Double` to reduce rounding error. Labels still need integer pixel positions because Swing text APIs use integer coordinates.

## Label Scale

Coordinate labels are divided by `10` before display. This keeps values compact enough for large maps.

For example, an internal coordinate of `400` is displayed as `40`.

The root origin displays only one `0` label. The y-axis label at the origin is skipped to avoid duplicate zero labels.

## Y Axis Direction

`showCoordinateAxisYInverted` controls y-axis label direction.

Default is `false`:

- Positive y labels are above the origin.
- Negative y labels are below the origin.

When enabled:

- Negative y labels are above the origin.
- Positive y labels are below the origin.

This is useful for historical charts and other visual layouts where y increases downward.

## Remaining Precision Notes

A small amount of visual mismatch can still happen because Freeplane node positions are produced by Swing layout at the current zoom, and many component sizes are integer based.

The coordinate grid itself avoids per-line integer rounding for line geometry, but node bounds, label positions, and layout calculations still contain integer conversions. Further precision work would likely require deeper changes in layout and node coordinate reporting.
