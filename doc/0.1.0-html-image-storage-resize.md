# 0.1.0 HTML 图片存储、去重与交互调整

## 背景

本轮改动围绕粘贴或拖入图片创建节点后的存储和交互体验。

原有实现把图片作为 external object 挂在节点上。这样会让图片节点和普通 HTML 文本节点形成两套模型，也不方便在 text/details 中统一编辑、复制、删除和调整图片尺寸。

新的方向是：图片保存为节点 text/details 的 HTML `<img>`，图片文件仍按存储策略落盘，但节点内容只保存 HTML 引用。

## 存储设计

图片存储策略通过 `image_storage_location` 控制：

- `current`：保存到当前脑图所在目录下的图片目录。
- `global`：保存到全局图片目录。
- `embedded`：保留为可内联的 data/source 形式。

图片保存时会基于内容做去重，避免重复粘贴同一张图片时生成多个相同文件。去重后的 `StoredImage` 会携带 source、fileName、width、height。HTML 插入时写入：

```html
<img src="..." width="..." height="..." src-width="..." src-height="...">
```

`width` / `height` 是当前显示尺寸，`src-width` / `src-height` 是源图尺寸。后续拖拽调整图片大小时，只改显示尺寸，源尺寸保留用于重置或继续按比例计算。

## 创建图片节点

粘贴剪贴板图片和拖入图片文件时，节点不再创建 external object，而是把图片插入节点 text HTML。

新建图片节点时，如果图片尺寸已知，会同步设置节点宽度，保证节点宽度不小于图片宽度，避免只显示图片的一小部分。

对于只有图片、没有文本的节点，拖拽图片边缘调整尺寸时，会实时同步节点宽度和图片宽度。

## 节点边缘拖拽

MainView 增加边缘拖拽交互：

- 左右边缘：调整节点宽度。
- 上下边缘：按比例调整节点字号，最小字号为 1，没有最大限制。
- 右下角：同时调整节点宽度和字号。
- 点击对应边缘：恢复对应设置。右下角点击同时恢复宽度和字号。

为了减少和节点移动拖拽的冲突，边缘触发区向外扩展，向内保持较窄。低缩放时如果视图尺寸仍足够，不再直接禁用节点宽度/字号拖拽；图片尺寸拖拽仍保留最小可操作尺寸阈值，避免小图或小缩放状态下误触。

## 图片边缘拖拽

text 和 details 中的图片都支持拖拽图片边缘同比例缩放。这里调整的是图片边缘，不是节点边缘。

图片命中检测走 Swing HTML renderer 的 view 树，并显式考虑 map zoom。之前定位不准的主要原因是 HTML view 的布局矩形和鼠标点不在同一坐标系。现在先用未缩放分配矩形查找图片，再缩放回组件坐标。

对 details 特别注意：details 的数据源必须按 API 语义取 raw HTML，即脚本 API 中的 `node.getDetailsText()`。实现上它对应：

```java
DetailModel.getDetailText(node)
```

不能用 `DetailsView.getText()` 或 `ZoomableLabel.getText()` 作为 details 的图片源，因为那是显示层文本，可能已经被 `TextController` transformer 加工，也可能被补了 `<base>`。右键菜单和图片拖拽都需要使用 raw details HTML 才能正确找到 `<img>` 并写回。

## 图片右键菜单

text/details 图片右键菜单增加：

- 打开
- 定位
- 删除引用
- 删除文件和引用

`删除引用` 只移除 text/details HTML 中对应的 `<img>`。

`删除文件和引用` 标红显示，只在图片 source 能解析为本地文件时启用。该动作先删除 HTML 引用，再尝试删除磁盘文件。文件删除失败时会提示错误，但不会回滚 HTML 引用删除。

删除操作按图片序号处理，避免多图节点中总是误删第一张图片。

## 菜单位置

`Format -> Image settings` 提升为和 `Node core` 同级的菜单项。`Image settings` 下保留 `Image storage location` 的 radio group。

## 关键实现点

- `ImageHtmlInserter` 负责创建、插入、按序号调整、按序号读取 source、按序号删除 HTML 图片。
- `HtmlImageHitDetector` 负责从 `ZoomableLabel` 的 HTML renderer 中定位鼠标命中的图片，并支持传入 raw HTML 作为真实图片源。
- `NodeSizeDragController` 处理 MainView 的节点宽度、字号和 text 图片拖拽。
- `DetailsImageDragController` 处理 details 图片拖拽，并使用 `DetailModel.getDetailText(node)` 作为 raw HTML。
- `HtmlImagePopupMenu` 处理 text/details 图片右键菜单，也按目标类型读取 node text 或 raw details HTML。
- `DetailsViewMouseListener` 先处理图片拖拽和图片右键，再回退到原 details 选择、编辑和节点 popup 行为。

## 放弃或修正过的方案

- 只把 `Image storage location` 从 `Image settings` 子菜单中提出来是不对的。正确需求是把 `Image settings` 整个提升到和 `Node core` 同级。
- details 图片命中不能依赖 `label.getText()`。这个值是显示层内容，不等价于 API 的 `node.getDetailsText()`。
- details 的鼠标事件不能让 map motion 逻辑先处理。图片交互需要优先消费事件，再让普通 map/details 行为作为 fallback。
- 右键删除图片文件不能只问“是否删除磁盘文件”。现在拆成两个显式按钮，避免语义不清。

## 后续风险

Swing HTML view 树对 `<img>` 的布局暴露有限。当前方案优先用 renderer view 树命中图片，必要时再用 raw HTML 的第一张图片尺寸兜底。复杂 HTML、多张图片、表格内图片等场景需要继续通过实际 UI 操作验证。

details 视图如果后续引入新的 transformer，把 details HTML 转成非 HTML icon 或其他组件，图片命中逻辑还需要继续对齐 raw `getDetailsText()` 与实际显示坐标。
