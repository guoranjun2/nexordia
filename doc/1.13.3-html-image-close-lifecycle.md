# HTML 图片脑图关闭卡顿排查与修复

## 背景

含大量 HTML 图片的脑图打开和浏览已经通过懒加载、按视口渲染、图片缓存优化得到改善，但关闭脑图仍然会在 macOS 上卡顿十几到几十秒。

现象特征：

- 图片主要出现在 MainView 和 DetailsView，二者都继承 ZoomableLabel。
- 折叠 details 后关闭明显变快。
- 折叠全部节点图片后关闭也不会卡。
- 卡顿期间 macOS AWT 日志反复出现 `CAccessibility.getFocusOwner`、`CAccessibility.getAccessibleIndexInParent`、`CInputMethod.selectedRange` 超时。

## 排查过程

第一轮怀疑 HTML renderer 生命周期，尝试在关闭时清理 `BasicHTML.propertyKey`、断开 renderer 子 view、清理 `TEXT_RENDERING_ICON`。这个方向单独不足以解决关闭卡顿。

第二轮怀疑 map 生命周期监听器，给 `beforeMapChanged`、`afterMapChanged`、`afterViewClose`、`onRemove` 等路径加分段计时。结果显示：

- 外层 `Controller.onRemove` 慢只是传播结果。
- 真正慢点在 `MapViewDockingWindows.afterViewClose`。
- 进一步拆分后发现 `DockingWindow.close()` 本身并不是根因。

第三轮尝试在 docking close 前替换 `MapViewScrollPane` 的 viewport view，结果破坏了 `MapViewScrollPane.MapViewPort` 的内部约定，触发 `ClassCastException: JPanel cannot be cast to MapView`，该方案废弃。

第四轮改为只替换 Infonode `View` 的顶层 content，避免 docking close 时携带完整脑图组件树。日志显示 `closeDockedWindow` 降到毫秒级，但耗时转移到了 `View.setComponent(new JPanel())`，说明真正卡点是 Swing 移除巨大组件树。

第五轮将 `MapView` root `NodeView` 的释放前移到 `MapViewController.remove()`，在 `afterMapViewClose` 之前释放节点组件树。继续拆分后发现：

- 业务清理很快。
- `NodeView.remove()` 递归逐个 `parent.remove(child)` 很慢。
- 改成只对 root 做一次 `mapView.remove(root)` 后仍然慢，说明单次 root removal 也会触发整棵子树 `removeNotify`，macOS Accessibility/InputMethod 会同步扫描含图片 HTML 的组件树。

最终确认：关闭卡顿主要来自带图片 HTML 的 `ZoomableLabel` 组件树在 `removeNotify` 阶段被 macOS Accessibility/InputMethod 同步访问。

## 最终方案

最终保留的修复点：

- `ScaledHTML.detachRenderer(JLabel)` 统一释放 Swing HTML renderer：
  - 清理 `BasicHTML.propertyKey`。
  - 断开旧 renderer 的子 view parent。
- `ZoomableLabel.releaseHtmlForMapClose()` 在关闭 map 前释放 label 上的 HTML 状态：
  - detach HTML renderer。
  - 清理 `TEXT_RENDERING_ICON`。
  - 清理 icon。
  - 清空 label text，避免 Accessibility 继续读取含 `<img>` 的 HTML 文本。
- `NodeView.releaseForMapClose()` 作为关闭脑图专用释放路径：
  - 递归释放子节点的 HTML label 状态。
  - 保留必要业务清理：`onViewRemoved`、`attributeView.viewRemoved`、`removeViewer`、`deselect`。
  - 不再对每个节点递归调用 `parent.remove(child)`。
- `MapViewController.remove()` 在 `afterMapViewClose` 前调用 root 释放：
  - 恢复临时 root 到真实 root。
  - 调用 `root.releaseForMapClose()`。
  - 最后只从 `MapView` 移除 root 一次。

## 验证数据

关键日志演进：

- 初始：`closeDockedWindow` 约 37500ms。
- 替换 Infonode View content 后：`closeDockedWindow` 降到约 4ms，但 `detachContent` 约 18200ms。
- 前移 root remove 后：`detachContent` 降到 0ms，但 `removeRoot` 约 18400ms。
- 跳过逐节点 Swing remove 后：`releaseRoot` 约 2ms，`removeRoot` 仍约 18700ms。
- 释放 MainView/DetailsView/note view 的 HTML renderer 和 text 后：
  - 图片脑图关闭约 286ms。
  - 另一张图片脑图关闭约 591ms。
  - 无图片脑图关闭约 62ms。

验证命令：

```bash
gradle :freeplane:compileJava
gradle :freeplane:test --tests 'org.freeplane.core.ui.components.html.*'
git diff --check
```

## 放弃的方案

- 替换 `MapViewScrollPane` 的 viewport view：破坏 `MapViewPort` 对 `MapView` 类型的假设。
- 仅清理 docking close 路径：只能转移耗时，不能解决 root 组件树移除。
- 清焦点、隐藏 `MapView`、暂停 repaint：对 `NodeView.remove()` 卡顿没有明显帮助。
- 仅跳过逐节点 `parent.remove(child)`：业务清理变快，但 root 子树 `removeNotify` 仍然会触发 macOS Accessibility 卡顿。

## 风险与后续

该方案只用于 map view 关闭路径，不改变普通折叠、展开、编辑、刷新时的 `NodeView.remove()` 行为。

后续可以补充 focused unit test：

- `ScaledHTML.detachRenderer` 后 `BasicHTML.propertyKey` 为空。
- `ZoomableLabel.releaseHtmlForMapClose` 后 text、icon、text rendering icon、HTML renderer 均被清理。
- `NodeView.releaseForMapClose` 不触发逐节点 Swing remove，同时清理 model viewer。
